## Issue

**F006 — Control panel shows no app icons**

The top control panel bar is empty — no app icons are rendered. In C++ Eagle Mode the panel displays a context-sensitive set of app icons (e.g. New Window, Fullscreen, Select Files, Help, Root, Chess, MetaWalk, Roleplay, Home, Virtual Cosmos, Mines, Clock at the Eagle/top level; a richer file-manager toolbar when zoomed into a directory).

**Repro:** Launch app. Observe the top control panel — no app icons are visible. In C++ Eagle Mode the same panel shows app icons at every zoom level.

## Phase 1 — Root Cause Investigation

**E1.1** (`crates/emmain/src/emBookmarks.rs:546-570`): `emBookmarkButton::Paint` shows
text name but never loads or displays the icon image. The `entry.Icon` field (e.g.
"chess.tga") is stored but never consumed by Paint.

**E1.2** (`crates/emmain/src/emBookmarks.rs:527-535`): `emBookmarkButton` struct has no
`icon` or `icon_loaded` field — no caching mechanism exists.

**E1.3**: No call to `emGetInsResImage` or equivalent exists anywhere in the Rust codebase
for bookmark icons. TGA icon loading from disk is not wired up to `emBookmarkButton`.

**E1.4** (golden test `crates/eaglemode/tests/golden/control_panel.rs:30-47`): The golden
test confirms the control panel IS creating "lMain" with a non-zero layout rect — so the
panel is rendering (Reality A), not empty (Reality B). This means the bug is visual
(missing icons on buttons), not a missing/empty panel.

**E1.5** (C++ `src/emMain/emBookmarks.cpp:1547-1591`): C++ `emBookmarkButton::Update`
loads the icon via `emTryGetResImage(GetRootContext(), abs_path)` where the path resolves
to `emGetInstallPath(EM_IDT_RES, "icons") / icon_name`. Then calls `SetIcon(img)` on the
emButton (emBorder). Rust emBookmarkButton has no equivalent update/icon-load flow.

**E1.6** (secondary, out of scope for this pass): `emMainControlPanel::create_children`
at `crates/emmain/src/emMainControlPanel.rs:187-211` never creates the ContentControlPanel
child (C++ `ContentView.CreateControlPanel(*this, "context")`). This is a separate
architectural issue requiring cross-tree communication — filed as F012.

**E1.7** (`crates/emcore/src/emResTga.rs:39`): `emcore::emResTga::load_tga` is public.
`emcore::emBookmarks::emBookmarksModel::GetDefaultIconDir()` returns the correct path
`<em>/res/icons`. The infrastructure exists to load TGA icons from disk.

**Root cause chain:**
1. User sees no app icons (Chess, Root, Home, Virtual Cosmos, Mines, Clock, etc.)
2. ← The "app icons" are bookmark buttons in `emBookmarksPanel` (lMain, right portion of control panel)
3. ← `emBookmarkButton::Paint` only draws a colored rect + text name — no icon image (E1.1)
4. ← `emBookmarkButton` has no icon loading/caching fields (E1.2, E1.3)
5. ← `emGetInsResImage` not implemented; bookmark icon TGA load not wired (E1.3, E1.5)

## Phase 2 — Pattern Analysis

**C++ reference behavior** (`emBookmarks.cpp:1547-1591`):
- `emBookmarkButton::Update()` calls `emTryGetResImage(ctx, GetAbsolutePath(icon, defaultIconDir))`
- Icon dir = `emGetInstallPath(EM_IDT_RES, "icons")` = `<em>/res/icons`
- Full path = `<em>/res/icons/<icon_name>` (e.g., `<em>/res/icons/chess.tga`)
- `SetIcon(img)` sets icon on emBorder; `SetIconAboveCaption()` places icon above caption
- `SetMaxIconAreaTallness(9.0/16.0)` limits icon area aspect ratio
- `emButton::Cycle` / `emBorder::Paint` renders icon in label area above caption text

**Rust difference:**
- `emBookmarkButton::Paint` draws colored rect + text only (E1.1)
- `emBookmarkRec.entry.Icon` field IS loaded from the rec file (e.g., "chess.tga")
- `emBookmarksModel::GetDefaultIconDir()` IS implemented and returns correct path (emBookmarks.rs:479-482)
- `emcore::emResTga::load_tga` IS public and can load TGA from `&[u8]` (emResTga.rs:39)
- `emPainter::paint_image_full` IS public and can paint scaled images (emPainter.rs:2043)
- Missing: the call chain from `entry.Icon` → disk load → `paint_image_full`

## Root Cause

`emBookmarkButton::Paint` has no icon-loading or icon-painting code. The `entry.Icon` field
(e.g., "chess.tga") is read from the bookmarks rec file but never used. C++ `emBookmarkButton::Update`
loads the icon via `emTryGetResImage` into an emBorder icon field, then emBorder::Paint renders it.
The Rust port has emBookmarksModel::GetDefaultIconDir() and emResTga::load_tga available but
these are never called from emBookmarkButton.

**Fix:** Add `icon_loaded: bool` and `icon: Option<emImage>` to `emBookmarkButton`. In `Paint`,
on first call, load TGA from `GetDefaultIconDir() / entry.Icon`. If successful, paint icon in
upper portion of button, text below (matching C++ SetIconAboveCaption behavior).

**Out of scope (F012):** ContentControlPanel wiring (context-sensitive icons when zoomed into
file manager) requires architectural design for cross-tree communication. Filed separately.
