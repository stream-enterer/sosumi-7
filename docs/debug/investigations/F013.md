## Phase 1 â€” Root Cause Investigation

**E1.1** (`crates/emfileman/src/emDirPanel.rs:525-537`): `emDirPanel::CreateControlPanel`
calls `parent_ctx.as_sched_ctx().expect("CreateControlPanel requires scheduler-reach PanelCtx")`
and then constructs `emFileManControlPanel::new(&mut sched, ctx)`. The expect is the
observed panic site.

**E1.2** (`crates/emfileman/src/emFileManControlPanel.rs:92`):
`emFileManControlPanel::new` has signature
`pub fn new<C: emcore::emEngineCtx::ConstructCtx>(cc: &mut C, ctx: Rc<emContext>)`.
Requires a `ConstructCtx` (i.e. `SchedCtx`/`InitCtx`) to register engines and
signals for the new control-panel widgets. Scheduler reach is genuinely required
here â€” not a defensive check.

**E1.3** (`crates/emcore/src/emEngineCtx.rs:560-574`): `PanelCtx::as_sched_ctx`
returns `Some(SchedCtx)` only when all five of `scheduler`, `framework_actions`,
`root_context`, `framework_clipboard`, `pending_actions` are `Some`. Any `None`
returns `None`.

**E1.4** (`crates/emcore/src/emEngineCtx.rs:468-482`): `PanelCtx::new(tree, id, tallness)`
sets all five handles to `None`. Only `PanelCtx::with_sched_reach` (lines 532-553)
attaches them.

**E1.5** (`crates/emcore/src/emPanelTree.rs:2109-2132`): `create_control_panel_in`
(the cross-tree variant added by F012 fix 851ca26b) builds the PanelCtx via
`PanelCtx::new(target_tree, parent_arg, current_pixel_tallness)` â€” the no-reach
constructor. This PanelCtx is then passed to the inner `behavior.CreateControlPanel`
call, so `parent_ctx.as_sched_ctx()` returns `None` and E1.1's `.expect` panics.

**E1.6** (`crates/emmain/src/emMainWindow.rs:807-881`, `ControlPanelBridge::Cycle`):
The production call site holds `&mut EngineCtx<'_>` which carries all five
scheduler-reach handles (`ctx.scheduler`, `ctx.framework_actions`,
`ctx.root_context`, `ctx.framework_clipboard`, `ctx.pending_actions`). The fields
are independent of `ctx.windows` so they can be split-borrowed alongside `win`.
Nothing in the current call site threads them into `create_control_panel_in`.

**E1.7** (`crates/emcore/src/emPanelTree.rs:2080-2102`): The non-cross-tree
`CreateControlPanel` has the same latent bug (uses bare `PanelCtx::new`) but
is only called from unit tests (`emPanelTree.rs:3065, 3078`) that use a
trivial `ControlCreator` behavior which does not need scheduler reach. Not
currently triggered in production.

**E1.8** (`crates/emfileman/src/emDirEntryPanel.rs:616-626`): Same
`as_sched_ctx().expect(...)` pattern â€” would panic identically if reached via
the same path. Any fix must cover emDirEntryPanel too (not just emDirPanel).

### Root cause chain

1. User zooms into File System panel â†’ content view's active panel changes â†’
   emView fires `control_panel_signal`.
2. `ControlPanelBridge::Cycle` (emMainWindow.rs:808) runs, walks to the active
   panel, and calls `content_sub_tree.create_control_panel_in(active, ctrl_sub_tree, ctrl_root, "context", tallness)`
   (emMainWindow.rs:855).
3. `create_control_panel_in` (emPanelTree.rs:2109) builds `PanelCtx::new(...)`
   without scheduler-reach handles (E1.4, E1.5).
4. The walk reaches `emDirPanel` as a parent-chain ancestor of the active panel.
   `emDirPanel::CreateControlPanel` calls `parent_ctx.as_sched_ctx().expect(...)`.
5. `as_sched_ctx` returns `None` because all five handles are `None` (E1.3) â†’
   `.expect` panics with the observed message.

## Phase 2 â€” Pattern Analysis

**E2.1** (`~/Projects/eaglemode-0.96.4/src/emCore/emView.cpp:193-199`):
C++ `emView::CreateControlPanel(emPanel &parent, const emString &name)` delegates
to `ActivePanel->CreateControlPanel(parent, name)`. `parent` is an `emPanel&`
with direct access to its `emContext` (which owns the scheduler). Any override
constructs the new control panel with `parent` as `ParentArg` â€” the constructor
extracts scheduler/context from `parent` via `emPanel::SetParent`. Scheduler
reach is implicit in C++ because every emPanel carries its emContext.

**E2.2** (`~/Projects/eaglemode-0.96.4/src/emCore/emPanel.cpp:1244-1247`):
`emPanel::CreateControlPanel` walks up the parent chain until a subclass overrides
and returns a non-null panel. The Rust `PanelTree::create_control_panel_in`
(emPanelTree.rs:2109-2132) mirrors this walk behaviorally â€” the only divergence
is *how* scheduler reach is conveyed.

**E2.3** (comparison): Rust's `PanelCtx` is the language-forced adaptation of
C++'s implicit "every emPanel has an emContext" access. Full scheduler reach
is conveyed through five handles: `scheduler`, `framework_actions`, `root_context`,
`framework_clipboard`, `pending_actions`. `PanelCtx::with_sched_reach`
(emEngineCtx.rs:533-553) is the full-reach constructor. `PanelTree::create_control_panel_in`
currently uses the no-reach `PanelCtx::new` â€” this is the observable difference
from C++ behavior (C++ never loses scheduler reach here).

### Phase 2 gate

C++ behavior confirmed: CreateControlPanel always executes with full context/scheduler
reach because emPanel carries emContext. Rust divergence: `create_control_panel_in`
throws the handles away. Fix is a below-surface ownership adaptation â€” thread
the handles through rather than constructing the PanelCtx bare.

## Root Cause

`PanelTree::create_control_panel_in` (emPanelTree.rs:2109) constructed the
inner `PanelCtx` via the no-reach `PanelCtx::new`. When the F012
ContentControlPanel wiring began calling this method from
`ControlPanelBridge::Cycle`, the parent-chain walk reached
`emDirPanel::CreateControlPanel`, which requires scheduler reach to
construct `emFileManControlPanel`. `parent_ctx.as_sched_ctx()` returned
`None` and `.expect(...)` panicked. Fix: thread the five scheduler-reach
handles through so the inner `PanelCtx` is built via
`PanelCtx::with_sched_reach`. C++ parity: every emPanel implicitly carries
its emContext (with scheduler); Rust conveys the same reach through explicit
handles.
