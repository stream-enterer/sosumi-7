## Issue

**F011 — emDirPanel and emDirStatPanel show 'No file model' — SetFileModel never called**

`emDirPanel::Paint` calls `file_panel.paint_status` when loading (`dir_model.is_some() && !loading_done`). `file_panel` is never connected to the model via `SetFileModel(Some(...))`, so `GetVirFileState() = NoFileModel` → renders dark-red "No file model" text.

Same gap in `emDirStatPanel` (documented in test `panel_initial_vfs_is_no_model`).

## Phase 1 — Root Cause Investigation

**E1**: `emDirPanel::Paint` (line 434-435): when `!loading_done && dir_model.is_some()`, calls `self.file_panel.paint_status`. With `file_panel.last_vir_file_state = NoFileModel` (never set), renders `rgba(128,0,0,192)` text "No file model" on canvas background.

**E2**: `emDirPanel::notice` (lines 400-406): when `keep_model=true && dir_model.is_none()`, sets `self.dir_model = Some(emDirModel::Acquire(...))` but never calls `self.file_panel.SetFileModel(Some(...))`.

**E3**: `emDirPanel::notice` (lines 407-413): when model cleared, calls `self.file_panel.SetFileModel(None)` — only the None path is ported.

**E4**: `emDirModel::get_file_state()` returns owned `FileState`. `FileModelState::GetFileState()` requires `&FileState`. So `emDirModel` needs a stored `state: FileState` field to implement `FileModelState`.

**E5**: `emDirStatPanel` same pattern — `file_panel.paint_status` called (line 131) but `SetFileModel(Some(...))` never called. Test line 224 explicitly asserts `NoFileModel` as initial state.

## Phase 2 — Pattern Analysis

**C++ emDirPanel::Notice** (emDirPanel.cpp):
```cpp
if (IsViewed() || GetSoughtName()) {
    if (!GetFileModel()) SetFileModel(emDirModel::Acquire(GetRootContext(), GetPath()));
} else if (GetFileModel()) {
    SetFileModel(NULL);
}
```
Rust port is missing the `SetFileModel(Some(...))` call.

**C++ emDirStatPanel**: extends `emFilePanel`. C++ `emDirStatPanel::Notice` calls `SetFileModel(emDirModel::Acquire(...))` when viewed.

## Phase 3 — Hypotheses

HYPOTHESIS: Adding `state: FileState` to `emDirModel`, implementing `FileModelState`, and calling `file_panel.SetFileModel(Some(...))` in `emDirPanel::notice`/Cycle when the model is acquired will change `file_panel.last_vir_file_state` from `NoFileModel` to `Waiting`→`Loading`→`Loaded`. / OUTCOME: to be confirmed by tests.

## Root Cause

`emDirPanel::notice` and `emDirPanel::Cycle` acquire `emDirModel` but never call `file_panel.SetFileModel(Some(...))`. The only `SetFileModel` call in Rust is `SetFileModel(None)` for cleanup. Blocked by: `emDirModel` lacks a `FileModelState` impl (no stored `FileState` field). Same gap in `emDirStatPanel`.

## Second Bug (found after Phase 4 — "Wait..." stuck after F011 fix)

**E6**: `emDirPanel::Cycle` holds `dm = dm_rc.borrow_mut()` while calling `self.file_panel.clear_custom_error()` / `set_custom_error()`. Both `dm_rc` (from `self.dir_model`) and `self.file_panel.model` are coerced clones of the same `Rc<RefCell<emDirModel>>` — they share the same `RefCell`. `compute_vir_file_state()` calls `model_rc.borrow()` → "RefCell already mutably borrowed" panic on the first Cycle call (Waiting→Loading transition). The panic unwinds through `PanelCycleEngine::Cycle` and `do_time_slice`, causing the engine to never re-run and the panel to stay at VirtualFileState::Waiting ("Wait..." yellow text).

**E7**: Same double-borrow bug in three other spots: Waiting/Err, Loading/Ok(true) (wrong drop order), and Loading/Err.

**Fix**: Add explicit `drop(dm)` before each `file_panel.*` call. Regression test `cycle_waiting_to_loading_advances_vir_state` verifies Waiting→Loading transition without panic.
