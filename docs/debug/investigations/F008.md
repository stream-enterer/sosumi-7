## Issue

**F008 — All app panels fail to load: dlopen error on libemFileMan.so**

Every app panel inside the cosmos (File System, Home, emStocks) fails to load with the error: "failed to load library 'libemFileMan.so': dlopen failed". This affects all panels that depend on native plugin libraries.

Repro: Launch app (release profile). Navigate into the cosmos. Click any app panel. Observe error.

## Phase 1 — Root Cause Investigation

**E1**: `libloading::Library::new("libemFileMan.so")` fails at runtime in release builds.

**E2**: Error format "failed to load library libemFileMan.so: ..." comes from `emFpPlugin.rs:578` (FpPluginError::LibraryLoad), which maps from LibError::LibraryLoad in `emTryOpenLib` at `emStd2.rs:298`.

**E3**: `ls target/release/libemFileMan.so` — file does not exist after `cargo build --release -p eaglemode`. The cdylib is only produced in `target/release/deps/libemFileMan.so`.

**E4**: The `eaglemode` release binary has `RUNPATH=$ORIGIN` (pointing to `target/release/`). `libemFileMan.so` is absent from `target/release/`, so dlopen fails.

**E5**: In debug builds: `libemFileMan.so` exists at `target/debug/libemFileMan.so` (produced by prior `cargo test` or `cargo build --workspace` runs that build dev-deps). This is why debug works but release fails.

**Root cause chain**: `emfileman` and `emstocks` were in `[dev-dependencies]` of `eaglemode`. Dev-deps are only compiled for `cargo test`, not for `cargo run --release`. When running the release binary, `RUNPATH=$ORIGIN` points to `target/release/` where neither `libemFileMan.so` nor `libemStocks.so` exists.

## Phase 2 — Pattern Analysis

**C++ reference**: In C++ Eagle Mode, plugins are pre-built `.so` files installed to `$EM_DIR/lib`. The app binary links against `libemCore.so` which has `RPATH=$EM_DIR/lib`. Plugins are always present in the install directory — the C++ build system (Makefile) builds all plugins as part of `make install`. There is no equivalent of "only dev builds include plugins."

**Rust difference**: The Rust port uses `cargo run --release` as the "run" command, which only builds `eaglemode` and its compile-time dependencies. Plugin crates (`emfileman`, `emstocks`) were only in `[dev-dependencies]`, so they are excluded from release builds.

**Secondary gap**: Even when plugins ARE built as deps, Cargo places the cdylib in `target/{profile}/deps/`, not `target/{profile}/`. The original RUNPATH ($ORIGIN) only covers the latter.

## Phase 3 — Hypotheses

HYPOTHESIS: `libemFileMan.so` is absent from `target/release/` because emfileman is only a dev-dependency, so `cargo run --release` never builds it. / OUTCOME: CONFIRMED / EVIDENCE: E3, E4 — `ls target/release/libemFileMan.so` after `cargo build --release -p eaglemode` returned "No such file or directory". Once moved to `[dependencies]`, the file appears in `target/release/deps/libemFileMan.so`.

HYPOTHESIS: RUNPATH=$ORIGIN is sufficient once emfileman is in [dependencies]. / OUTCOME: RULED OUT — the cdylib lands in `deps/` not the profile root, so $ORIGIN alone misses it. / EVIDENCE: after moving to [dependencies] and building release, `target/release/libemFileMan.so` still absent; only `target/release/deps/libemFileMan.so` present.

## Root Cause

Two compounding issues:
1. `emfileman` and `emstocks` were in `[dev-dependencies]`, so they were never built for release profiles.
2. When added to `[dependencies]`, Cargo places their cdylib output in `target/{profile}/deps/`, not the profile root. The original RUNPATH (`$ORIGIN`) only searched the profile root.

**Fix**:
- Move `emfileman` and `emstocks` to `[dependencies]` in `crates/eaglemode/Cargo.toml`. This ensures all profiles build the plugin cdylibs.
- Extend RUNPATH in `build.rs` from `$ORIGIN` to `$ORIGIN:$ORIGIN/deps`. This ensures the binary finds the cdylib wherever Cargo places it (profile root for workspace builds, `deps/` for single-package builds).

New plugins follow the same convention: add to `[dependencies]` in `eaglemode/Cargo.toml`.

## Iteration 2 — Post-Verification Findings (2026-04-23)

**Manual verification result**: After the dlopen fix, panels show (a) a small dark-red box at far zoom and (b) black interior at closer zoom.

**E6**: Dark-red box = `VirtualFileState::NoFileModel` rendered by `emFilePanel::paint_status`. `emFileLinkPanel.file_panel` starts with `NoFileModel` because `set_link_model` never calls `file_panel.SetFileModel`. Before AutoExpand creates the child panel, `!is_good() && child_panel.is_none()` = true → paint_status renders `rgba(128, 0, 0, 192)`. Once AutoExpand runs and `child_panel.is_some()`, the condition clears and the border renders normally. Transient: only visible before first AutoExpand. Root cause: `emRecFileModel<T>` does not implement `FileModelState` → filed as F009.

**E7**: Black interior = `DirContentColor: 0x000000FF` (default theme). When emDirEntryPanel paints its content area background for a directory, it fills with pure black. This is expected Eagle Mode behavior (dark file-manager background). Not a bug.

**E8**: `Home.emFileLink` had `Path="~"`. In Rust, `PathBuf::join("~")` does NOT expand the tilde — results in `VcItemFiles/~` (nonexistent). C++ Eagle Mode does not ship a Home.emFileLink at all. Fix: changed to `BasePathType="home"` which calls `emGetInstallPath(Home)` = `$HOME`. Tests confirm path now resolves to `/home/alex`.

**Status**: F008 remains `needs-manual-verification`. The dlopen error is fixed. Home.emFileLink path is fixed. The dark-red transient state (F009) is a separate gap in `emRecFileModel` not implementing `FileModelState`.
