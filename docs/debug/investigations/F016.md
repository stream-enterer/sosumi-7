# F016 — `visit_fullsized` against an inner sub-view target races against `LayoutChildren`

## Summary

`emView::VisitFullsizedByIdentity` (and through it the user-visible
`visit_fullsized` control-channel command) targeting a panel inside a
sub-view does not zoom when the target panel's parent has not yet run
`LayoutChildren`. The visiting animator finds the target by identity,
but the target's `layout_*` fields are still at the
emPanel-default placeholder `(-2, -2, 1, 1)`. The animator's
goal-distance computation against this placeholder evaluates to
"already at goal" within one or two cycles, the animator returns
false, and the view never moves.

The bug is racy: when the binary spawns slowly (cold cargo build), the
sub-view's wrapper engine has time to cycle and run `LayoutChildren`
before the user's `visit_fullsized` arrives, the cosmos has a real
`(0, 0, 1, 1)` layout, and the visit succeeds. When the binary spawns
quickly (warm build), `visit_fullsized` arrives first and locks in a
no-op trajectory against the placeholder layout.

There is also an unrelated-but-symptomatic *startup* visit:
`emMainWindow`-or-equivalent fires a `VisitFullsized` against the
cosmos identity `:` at sub-view init. That visit hits the same bug
and silently fails for the same reason. The user does not perceive
this because the outer view has its own visit running at startup
which the user does see.

## Evidence

(temporarily added; removed in this commit). Sequence: `wait_idle` →
`visit root:content view` → `wait_for panel_viewed` → `wait_idle` →
`visit_fullsized view="root:content view" identity=":"` → `wait_for
panel_fullsized identity=":"` (10s timeout). Run with
`EMCORE_DLOG=1`.

Key facts:

- `panels_after=0` — the animator's identity walk reached the cosmos.
  Hypothesis #3 (identity mismatch) is **disproved**.
- `layout=(-2.0000,-2.0000,1.0000,1.0000)` — the
  emPanel default-constructed layout. `LayoutChildren` has not run
  on the cosmos's parent.
- `target=(0.0000,0.0000,1.8343)` — `CalcVisitFullsizedCoords` (in
  `crates/emcore/src/emView.rs:1607-1648`) returns `(relX, relY,
  relA)` derived from the panel's `GetHeight()` attribute and the
  view's `HomeWidth/HomeHeight/HomePixelTallness`. It is **not**
  affected by the panel's layout. Hypothesis #1 (wrong frame) is
  **disproved**: the goal in cosmos's local-rel coords is correct.
- The animator transitions `Curve → Direct → animate->false` in two
  cycles (~116 ms). On the third cycle the engine sleeps. By the time
  `LayoutChildren` runs (later) and the cosmos's layout becomes
  meaningful, the animator is already inactive.
- The eventual post-visit dump shows the cosmos at
  `Layout XYWH 0.499..,0.272..,0.0002..,0.0002..` — a sub-pixel speck.
  This is the parent's `LayoutChildren` finally running but
  computing a degenerate layout for a cosmos whose content engine
  also raced against the visit and never got to set up a real
  natural-size hint. (This sub-pixel cosmos rect is what makes the
  F015 `wait_for panel_viewed` predicate return true; see Bug B in
  the F016 plan.)

Slow-path observation (run after a fresh `cargo build` rebuild,
roughly +6s of compile time before binary launch): the cosmos zooms
visibly and `wait_for panel_fullsized` returns `ok:true`. This is
consistent with `LayoutChildren` having run before `visit_fullsized`.

## Root cause

Hypothesis **#2** from the F016 plan, with the specific mechanism:
the visiting view animator commits to a goal computed against a
placeholder layout when its target panel's parent has not yet run
`LayoutChildren`. The animator is not aware that the target panel's
position is meaningless; it treats `(-2, -2, 1, 1)` as a valid
ancestor-layout context and pieces together a no-op trajectory.

The layered C++ design protects against this in practice because
`LayoutChildren` runs synchronously during scheduler cycles before
animator cycles in the same tick — but the Rust port's sub-view
wrapper engine ordering does not currently guarantee that. (See
F010 / F011 history for the structural sub-view wrapper-engine work.)

## Phase 4 fix surface

Two viable shapes:

- **(a) Animator-side defense.** In
  `emVisitingViewAnimator::get_nearest_existing_panel`
  (`crates/emcore/src/emViewAnimator.rs`, around the dlog!-emitted
  call site), detect the placeholder layout on the resolved target's
  parent chain. If any ancestor has the
  default `(-2, -2, 1, 1)` layout AND has a parent that should have
  laid it out, return `None` (which transitions the animator to
  `GivingUp` for ≤1.5s — long enough for `LayoutChildren` to run).
  This is robust against any future racy-layout source.
- **(b) Caller-side ordering.** Ensure the inner sub-tree's
  `LayoutChildren` cascade has run before `VisitFullsizedByIdentity`
  is dispatched. Likely surface: the SVP's wakeup path
  (`emSubViewPanel`) or `emCtrlSocket::handle_visit_fullsized` could
  drain a layout pass first.

(a) is the safer choice because the bug surfaces both via the user's
control-channel call AND via the startup auto-visit (whose timing is
not under test control). It also makes the animator robust against
late-arriving layout invalidations, which matches the C++ behavior
that the bug observably violates.

## Closed hypotheses

- **#1 (goal in wrong frame of reference):** *disproved.*
  `CalcVisitFullsizedCoords` does not depend on layout; its output
  for cosmos was reasonable (`relA=1.83`).
- **#3 (identity `:` resolves wrong):** *disproved.* Animator's
  `panels_after=0` confirms the cosmos was found by identity walk.

## Phase 4 post-mortem (2026-04-25, this session)

**The Phase 3 root-cause statement above is incomplete.** Attempting
to fix it via the cited mechanism did not unbreak observable
startup-zoom parity with C++. Specifically:

### What was attempted

1. **Wake the inner UpdateEngine when panels are created in the
   inner sub-tree.** StartupEngine state 5/6 was passing
   `None` for the scheduler to `fire_init_notices` /
   `create_child`, so the inner sub-tree's `update_engine_id` was
   set but never woken. Patched both call sites to pass
   `Some(*scheduler)` via split-borrow on `EngineCtx`.
   `cargo nextest run` 2796/2796 passed.
2. **Live verification, no display:** spawned binary, drove via
   control channel, dumped, inspected dlog. Cosmos still rendered at
   sub-pixel `Layout XYWH 0.499..,0.272..,0.0002..,0.0002..`.
   `viewed_xywh = (642.32, 350.10, 0.354, 0.354)`. No visible zoom.
3. **User runtime verification:** "NO zoom whatsoever" on `cargo run
   --bin eaglemode`. Patch reverted.

### What further dlogging revealed

With dlog at every `RawVisit`, `UpdateChildrenViewing`, and
`SetGeometry` call site, plus `Backtrace::force_capture()` on the
suspect call:

- The animator's `RawVisit(cosmos, 0, 0, 1.83)` succeeded:
  `cosmos.viewed_xywh = (292.22, 0.00, 700.56, 700.56)` (cosmos
  centered, fullsize, in inner view's screen rect).
- **Immediately after**, `StartupEngine::Cycle` state 10 called
  `svp.raw_zoom_out(false, &mut sc)` — backtrace pinned to
  `crates/emmain/src/emMainWindow.rs:762`. This invokes
  `emView::RawZoomOut` → `RawVisit(sub_root, 0, 0, 1.0)`, which
  resets the inner view to its home rect. The cosmos's
  `viewed_xywh` is then recomputed as
  `sub_root.viewed × cosmos.layout = (642, 350, 0.23, 0.23)`.
- This is intentional: C++ `emMainWindow.cpp:455-465` does the
  exact same `RawZoomOut` at state 10. So matching C++ here is not
  a port bug.

### The real port-fidelity gap

Rust matches C++'s state 10 zoom-out, but **fails to perform C++'s
state 11 final `Visit(VisitIdentity, ...)`** that re-zooms into
the user's bookmark target. C++ `emMainWindow.cpp:466-484` (default
case after state 10): waits 100ms, then if `VisitValid`, calls
`ContentView.Visit(VisitIdentity, VisitRelX, VisitRelY,
VisitRelA, VisitAdherent, VisitSubject)`. That is the visible
startup zoom-in users observe.

User confirmation (2026-04-25): C++ binary on this machine zooms
to the homedir bookmark on startup; Rust binary does not. User has
a bookmark `LocationIdentity = ":"` with `LocationRelA = 0.00621`
(deep zoom-in) in `~/.eaglemode/emMain/bookmarks.rec`.

### Status of hypothesis #2

Hypothesis #2 (placeholder-layout cascade) describes a real
phenomenon visible in the dlog: the cosmos's parent does run its
`LayoutChildren` after `RawVisit` with the cosmos's natural sub-
pixel layout. But this is **not the controlling cause** of the
user-observable startup-zoom gap. Whether or not the inner
animator commits to a placeholder layout in the early states, the
state-10 zoom-out reverses any earlier zoom; the visible end-state
is determined by the state-11 `Visit` call, which is the actual
divergence point.

### Open

- Why does Rust's StartupEngine not produce the C++ state-11 visible
  re-zoom? Three sub-hypotheses, none yet tested:
  (a) `visit_valid` is never set true in Rust state 4
  (`SearchStartLocation` returns nothing or its result isn't
  applied);
  (b) state 9 sets the goal but the animator doesn't run / wake
  / activate;
  (c) the default-case state 11+ is not reached at all.
- These need targeted dlog at StartupEngine state entry plus
  `visit_valid`/`visit_identity`/animator-active before any further
  fix work. **Phase 4 stays open until this is resolved.**
