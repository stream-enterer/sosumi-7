## Phase 1 — Root Cause Investigation (original)

**E1**: `emFileManThemeData::default()` has `DirContentW: 0.0`, `DirContentH: 0.0`. These stay zero if the theme file never loads. `update_content_panel` lays out the child with `cw=0, ch=0` → invisible.

**E2**: `emFileManViewConfig::Acquire` fell back to `emFileManTheme::Acquire(ctx, "default")` when the config had no theme name. No `default.emFileManTheme` exists → `TryLoad` fails → all dimensions stay 0.0.

**E3**: In C++, `emFileManConfig` constructor (lines 75-79) validates `ThemeName` against `emFileManThemeNames` and replaces it with `GetDefaultThemeName()` (= "Glass1") if invalid. The Rust `emFileManConfig::Acquire` was missing this step entirely.

**E4**: C++ `emFileManViewConfig::emFileManViewConfig` (line 278) simply does `Theme=emFileManTheme::Acquire(GetRootContext(),ThemeName)` — no fallback needed because `emFileManConfig` already guaranteed a valid name.

## Phase 2 — Pattern Analysis

**C++ reference**: `emFileManConfig` constructor acquires `emFileManThemeNames`, checks `IsExistingThemeName`, falls back to `GetDefaultThemeName()`, saves. `emFileManViewConfig` trusts the result — no fallback.

**Rust divergence (resolved by prior fix)**: `emFileManConfig::Acquire` had no theme-name validation. `emFileManViewConfig::Acquire` had an incorrect `"default"` fallback that pointed to a nonexistent theme.

### Phase 3 — HYPOTHESIS E CONFIRMED 2026-04-24 pm (code analysis)

**E3.5 — C++ reference:** `emViewAnimator::Activate` at
`~/Projects/eaglemode-0.96.4/src/emCore/emViewAnimator.cpp:68-84`
calls `WakeUp()` on line 81 to enqueue itself on the scheduler. In C++,
`emViewAnimator` derives from `emEngine`, so the animator IS the engine
and `WakeUp()` is a base-class method that pushes the engine onto the
scheduler's busy list. `emVisitingViewAnimator::Activate`
(`emViewAnimator.cpp:1040-1044`) chains directly to
`emViewAnimator::Activate`, so visiting also wakes.

**E3.6 — Rust divergence:** `emVisitingViewAnimator::Activate`
(`crates/emcore/src/emViewAnimator.rs:896-898`) is two lines:
```rust
pub fn Activate(&mut self) {
    self.active = true;
}
```
It only sets a flag; it does NOT wake the wrapper engine. The Rust port
splits the animator from the engine: the animator is a plain type, and
`VisitingVAEngineClass` (`emView.rs:267-353`) is the registered engine
that wraps it. The animator does not hold a reference to the wrapper
engine's `EngineId`, so it cannot wake it from `Activate`.

**E3.7 — Wrapper engine is registered asleep:**
`EngineScheduler::register_engine` (`emScheduler.rs:313-327`) sets
`awake_state: -1` (sleeping). `RegisterEngines` (`emView.rs:3267-3308`)
registers `VisitingVAEngineClass` but does NOT call `wake_up` on its
`engine_id`. Compare with `UpdateEngineClass`, which IS woken at line
3307 via `WakeUpUpdateEngine(ctx)`.

**E3.8 — No production caller wakes the visiting engine:** Searching
the production tree (`grep visiting_va_engine_id`) finds the field
declared at `emView.rs:462`, set inside `RegisterEngines`, and read
only inside test paths and `take()` calls during teardown. The only
production code path that could wake it is the wrapper engine's own
`Cycle` (which self-rewakes by returning the result of `va.animate`,
per `emScheduler.rs:666-674`) — but `Cycle` only runs if the engine is
already awake, and it never gets the initial kick.

**E3.9 — Why outer-view zoom still works:** The user's mouse-wheel zoom
on the outer view goes through `emView::Zoom` (`emView.rs:1296-…`),
which calls `RawVisit` directly without using `VisitingVA`. So outer
zoom works for a different reason: it bypasses the animator. Any
*programmatic* `VisitPanel`/`VisitFullsized` (StartupEngine's
`VisitFullsized(":")` on the inner view, control-channel `visit`, focus
adjustments) goes through the animator and is silently dropped.

**E3.10 — Existing test masks the bug:** The regression test
`visiting_va_cycles_when_activated` (`emView.rs:7398-7474`) calls
`SetGoal + Activate` then `sched.borrow_mut().wake_up(visiting_id);`
(line 7454) before `DoTimeSlice`. The manual wake_up is the test
working around the missing wake in `Activate`. Without that line, the
animator would never cycle.

**Outcome:** HYPOTHESIS E (`VisitPanel/VisitFullsized doesn't drive the
sub-view animator on this Rust port`) is CONFIRMED. The defect is
narrower than "sub-view-specific" — `VisitingVA::Activate` fails to
wake the engine on *both* outer and inner views. Outer view escapes the
symptom because user mouse-zoom uses the synchronous `RawVisit` path.

EVIDENCE: E3.1 (smoking-gun runtime capture), E3.5 (C++ reference),
E3.6 (Rust code), E3.7 (registration sleeps), E3.8 (no production wake
path), E3.9 (outer-view explanation), E3.10 (test workaround).

   does `emSubViewPanel::sub_view_and_tree_mut().0` register its

   a per-window/per-sub-view scheduler? If different, the wake call

b. **Wake livelock:** The 8-min 98% CPU is consistent with
   `VisitingVAEngineClass::Cycle` self-rewaking each slice without
   making progress. C++ `emEngine::WakeUp()` returning when already
   awake is a common base case; check `EngineScheduler::wake_up_engine`
   semantics for re-entry. Also check whether `va.animate` can return
   a "still active" indication that re-queues the engine for the next
   slice forever (e.g. animation curve never reaches the goal because
   the goal coords are bogus, or `is_active()` doesn't clear).

`SetGoalFullsized` only sets `state = Curve` via `activate_goal`;
`self.active = true` is `Activate()`'s job. With `self.active`
false, `SVP::Cycle`'s `anim.animate(...)` early-returns at
`if !self.active` (`emViewAnimator.rs:2117`) and the animator is
dropped.

State 9 (`-visit` CLI arg path, `emMainWindow.cpp:439-454`) had
the same omission.

**E5.7 — wake gap.** Even with `Activate()` added, the SVP's own
`PanelCycleEngine` adapter wasn't waking. Second test run after
adding only `Activate()` (`/tmp/F010_dlog_after_fix.stderr`)
showed identical "no animate dlog for inner" symptom. C++ collapses
animator and engine into one type so `Activate→WakeUp` wakes the
cycle implicitly; Rust splits them, requiring an explicit wake on
the host engine after installing the animator.

**E6.4 — Throttle is per-call one-entry in BOTH ports.**
`crates/emfileman/src/emDirPanel.rs:361` calls `dm.try_continue_loading()`
exactly once per Cycle. `crates/emcore/src/emFileModel.rs:365-417`'s
`step_loading` also calls `try_continue_loading` exactly once per call
(no inner budget loop) — `emDirPanel` simply bypasses `step_loading`
and drives the model directly. C++ `StepLoading`
(`emFileModel.cpp:340-398`) likewise calls `TryContinueLoading()`
exactly once per call. C++ `Load(immediately)` (`emFileModel.cpp:111-129`)
loops only when `immediately==true`; the per-Cycle path advances one
entry per call in both ports.

**Therefore the slow-loading cause is NOT the throttle structure.**
The earlier explore-agent claim that C++ `StepLoading` loops per
budget was incorrect. Slow-loading cause is unidentified; candidates:

a. **Cycle frequency divergence** — does the Rust scheduler invoke
   `emDirPanel::Cycle` at a lower rate than C++? `emDirPanel.rs:399-401`
   does `return true` when loading, which should re-queue the engine,
   but the actual re-run cadence under the Rust scheduler vs C++'s
   needs measurement. Sub-question: is there a frame-timer gating the
   re-wake?
b. **Per-call cost** — `emDirEntry::from_parent_and_name` does one
   `lstat`. Compare against C++ `emDirEntry::Load` for any extra work
   (`String` allocation, `to_str().unwrap_or` fallback, etc.).
c. **Bypass of `step_loading`** — even if `step_loading` is also
   one-per-call, going through it sets `memory_need` and updates
   `progress` differently. Verify whether routing through it changes
   observable cadence.

This deserves its own narrow investigation pass — likely a separate
issue (`F017`).
