## Issue

**F010 — Directory listing never loads inside emDirEntryPanel — stays black indefinitely**

When zooming into a cosmos file-manager panel (FS, Home), the content area renders as a black rectangle and never populates with a directory listing at any zoom level. Expected: `emDirEntryPanel::update_content_panel` creates an `emDirPanel` child when zoomed in past `MinContentVW`. Observed: content area stays black all the way.

Repro: Launch app. Navigate into cosmos. Zoom into File System or Home panel all the way. Interior stays black — no directory listing appears.

2. [x] Check default theme values — `DirContentW: 0.0`, `DirContentH: 0.0`, `MinContentVW: 0.0` in default()
3. [x] Read C++ emFileManConfig constructor — theme name validation lives there

## Phase 1 — Root Cause Investigation

**E1**: `emFileManThemeData::default()` has `DirContentW: 0.0`, `DirContentH: 0.0`. These stay zero if the theme file never loads. `update_content_panel` lays out the child with `cw=0, ch=0` → invisible.

**E2**: `emFileManViewConfig::Acquire` fell back to `emFileManTheme::Acquire(ctx, "default")` when the config had no theme name. No `default.emFileManTheme` exists → `TryLoad` fails → all dimensions stay 0.0.

**E3**: In C++, `emFileManConfig` constructor (lines 75-79) validates `ThemeName` against `emFileManThemeNames` and replaces it with `GetDefaultThemeName()` (= "Glass1") if invalid. The Rust `emFileManConfig::Acquire` was missing this step entirely.

**E4**: C++ `emFileManViewConfig::emFileManViewConfig` (line 278) simply does `Theme=emFileManTheme::Acquire(GetRootContext(),ThemeName)` — no fallback needed because `emFileManConfig` already guaranteed a valid name.

## Phase 2 — Pattern Analysis

**C++ reference**: `emFileManConfig` constructor acquires `emFileManThemeNames`, checks `IsExistingThemeName`, falls back to `GetDefaultThemeName()`, saves. `emFileManViewConfig` trusts the result — no fallback.

**Rust divergence**: `emFileManConfig::Acquire` had no theme-name validation. `emFileManViewConfig::Acquire` had an incorrect `"default"` fallback that pointed to a nonexistent theme.

## Phase 3 — Hypotheses

HYPOTHESIS: Adding theme-name validation in `emFileManConfig::Acquire` (matching C++ constructor lines 75-79) and removing the `"default"` fallback from `emFileManViewConfig::Acquire` will cause the theme to load with real dimensions. / OUTCOME: CONFIRMED / EVIDENCE: E3, E4 — `theme_has_nonzero_dir_content_dimensions` test passes; 2687 tests pass.

## Root Cause

`emFileManConfig::Acquire` was missing the C++ theme-name validation step. With an empty config (fresh start), `theme_name` stays `""`. `emFileManViewConfig::Acquire` used `"default"` as the fallback instead of `GetDefaultThemeName()`. No `default.emFileManTheme` exists → theme load fails → all layout dimensions are 0.0 → content panel has zero size → directory listing is never visible.

**Fix**:
1. `emfileman/src/emFileManConfig.rs`: After `TryLoadOrInstall`, acquire `emFileManThemeNames`, validate `theme_name` with `IsExistingThemeName`, fall back to `GetDefaultThemeName()` and `Save()`. Mirrors C++ lines 75-79.
2. `emfileman/src/emFileManViewConfig.rs`: Remove `"default"` fallback; pass `tn` directly to `emFileManTheme::Acquire`. Mirrors C++ line 278.
