## Phase 1 ŌĆö Root Cause Investigation (original)

**E1**: `emFileManThemeData::default()` has `DirContentW: 0.0`, `DirContentH: 0.0`. These stay zero if the theme file never loads. `update_content_panel` lays out the child with `cw=0, ch=0` ŌåÆ invisible.

**E2**: `emFileManViewConfig::Acquire` fell back to `emFileManTheme::Acquire(ctx, "default")` when the config had no theme name. No `default.emFileManTheme` exists ŌåÆ `TryLoad` fails ŌåÆ all dimensions stay 0.0.

**E3**: In C++, `emFileManConfig` constructor (lines 75-79) validates `ThemeName` against `emFileManThemeNames` and replaces it with `GetDefaultThemeName()` (= "Glass1") if invalid. The Rust `emFileManConfig::Acquire` was missing this step entirely.

**E4**: C++ `emFileManViewConfig::emFileManViewConfig` (line 278) simply does `Theme=emFileManTheme::Acquire(GetRootContext(),ThemeName)` ŌĆö no fallback needed because `emFileManConfig` already guaranteed a valid name.

## Phase 2 ŌĆö Pattern Analysis

**C++ reference**: `emFileManConfig` constructor acquires `emFileManThemeNames`, checks `IsExistingThemeName`, falls back to `GetDefaultThemeName()`, saves. `emFileManViewConfig` trusts the result ŌĆö no fallback.

**Rust divergence (resolved by prior fix)**: `emFileManConfig::Acquire` had no theme-name validation. `emFileManViewConfig::Acquire` had an incorrect `"default"` fallback that pointed to a nonexistent theme.

### Phase 3 ŌĆö HYPOTHESIS E CONFIRMED 2026-04-24 pm (code analysis)

**E3.5 ŌĆö C++ reference:** `emViewAnimator::Activate` at
`~/Projects/eaglemode-0.96.4/src/emCore/emViewAnimator.cpp:68-84`
calls `WakeUp()` on line 81 to enqueue itself on the scheduler. In C++,
`emViewAnimator` derives from `emEngine`, so the animator IS the engine
and `WakeUp()` is a base-class method that pushes the engine onto the
scheduler's busy list. `emVisitingViewAnimator::Activate`
(`emViewAnimator.cpp:1040-1044`) chains directly to
`emViewAnimator::Activate`, so visiting also wakes.

**E3.6 ŌĆö Rust divergence:** `emVisitingViewAnimator::Activate`
(`crates/emcore/src/emViewAnimator.rs:896-898`) is two lines:
```rust
pub fn Activate(&mut self) {
    self.active = true;
}
```
It only sets a flag; it does NOT wake the wrapper engine. The Rust port
splits the animator from the engine: the animator is a plain type, and
`VisitingVAEngineClass` (`emView.rs:267-353`) is the registered engine
that wraps it. The animator does not hold a reference to the wrapper
engine's `EngineId`, so it cannot wake it from `Activate`.

**E3.7 ŌĆö Wrapper engine is registered asleep:**
`EngineScheduler::register_engine` (`emScheduler.rs:313-327`) sets
`awake_state: -1` (sleeping). `RegisterEngines` (`emView.rs:3267-3308`)
registers `VisitingVAEngineClass` but does NOT call `wake_up` on its
`engine_id`. Compare with `UpdateEngineClass`, which IS woken at line
3307 via `WakeUpUpdateEngine(ctx)`.

**E3.8 ŌĆö No production caller wakes the visiting engine:** Searching
the production tree (`grep visiting_va_engine_id`) finds the field
declared at `emView.rs:462`, set inside `RegisterEngines`, and read
only inside test paths and `take()` calls during teardown. The only
production code path that could wake it is the wrapper engine's own
`Cycle` (which self-rewakes by returning the result of `va.animate`,
per `emScheduler.rs:666-674`) ŌĆö but `Cycle` only runs if the engine is
already awake, and it never gets the initial kick.

**E3.9 ŌĆö Why outer-view zoom still works:** The user's mouse-wheel zoom
on the outer view goes through `emView::Zoom` (`emView.rs:1296-ŌĆ”`),
which calls `RawVisit` directly without using `VisitingVA`. So outer
zoom works for a different reason: it bypasses the animator. Any
*programmatic* `VisitPanel`/`VisitFullsized` (StartupEngine's
`VisitFullsized(":")` on the inner view, control-channel `visit`, focus
adjustments) goes through the animator and is silently dropped.

**E3.10 ŌĆö Existing test masks the bug:** The regression test
`visiting_va_cycles_when_activated` (`emView.rs:7398-7474`) calls
`SetGoal + Activate` then `sched.borrow_mut().wake_up(visiting_id);`
(line 7454) before `DoTimeSlice`. The manual wake_up is the test
working around the missing wake in `Activate`. Without that line, the
animator would never cycle.

**Outcome:** HYPOTHESIS E (`VisitPanel/VisitFullsized doesn't drive the
sub-view animator on this Rust port`) is CONFIRMED. The defect is
narrower than "sub-view-specific" ŌĆö `VisitingVA::Activate` fails to
wake the engine on *both* outer and inner views. Outer view escapes the
symptom because user mouse-zoom uses the synchronous `RawVisit` path.

EVIDENCE: E3.1 (smoking-gun runtime capture), E3.5 (C++ reference),
E3.6 (Rust code), E3.7 (registration sleeps), E3.8 (no production wake
path), E3.9 (outer-view explanation), E3.10 (test workaround).
