## Phase 1 — Root Cause Investigation (original)

**E1**: `emFileManThemeData::default()` has `DirContentW: 0.0`, `DirContentH: 0.0`. These stay zero if the theme file never loads. `update_content_panel` lays out the child with `cw=0, ch=0` → invisible.

**E2**: `emFileManViewConfig::Acquire` fell back to `emFileManTheme::Acquire(ctx, "default")` when the config had no theme name. No `default.emFileManTheme` exists → `TryLoad` fails → all dimensions stay 0.0.

**E3**: In C++, `emFileManConfig` constructor (lines 75-79) validates `ThemeName` against `emFileManThemeNames` and replaces it with `GetDefaultThemeName()` (= "Glass1") if invalid. The Rust `emFileManConfig::Acquire` was missing this step entirely.

**E4**: C++ `emFileManViewConfig::emFileManViewConfig` (line 278) simply does `Theme=emFileManTheme::Acquire(GetRootContext(),ThemeName)` — no fallback needed because `emFileManConfig` already guaranteed a valid name.

## Phase 2 — Pattern Analysis

**C++ reference**: `emFileManConfig` constructor acquires `emFileManThemeNames`, checks `IsExistingThemeName`, falls back to `GetDefaultThemeName()`, saves. `emFileManViewConfig` trusts the result — no fallback.

**Rust divergence (resolved by prior fix)**: `emFileManConfig::Acquire` had no theme-name validation. `emFileManViewConfig::Acquire` had an incorrect `"default"` fallback that pointed to a nonexistent theme.

- HYPOTHESIS B: `emDirPanel::Cycle` reaches `Loaded` but `update_children` either (i) runs with `visible_count == child_count == 0` (model done but entries empty), or (ii) runs and creates children, but `LayoutChildren` sizes them such that they clip off the viewed rect. / OUTCOME: RULED OUT — new test `real_stack_dir_panel_children_created_with_nonzero_rects_after_load` drives the full real_stack pipeline (emDirEntryPanel → emDirFpPlugin → emDirPanel) against a populated temp directory through DoTimeSlice until loading finishes, then asserts (i) child count == expected entries and (ii) every child's layout_rect is non-zero. Test passes — in headless harness the inner emDirPanel creates entries with valid rects. / EVIDENCE: `crates/emfileman/src/emDirPanel.rs:1356-1514` (new test); baseline theme geometry `DirContentW=0.546, DirContentH=0.205` gives `panel tallness = 0.375 = Height`, so `compute_grid_layout` produces valid cell rects.
- HYPOTHESIS D-visibility (scaled variant of B): at scale, one-entry-per-Cycle loading interacts with visibility such that partial state yields zero-rect children, or the final LayoutChildren overruns and collapses entries at high N. / OUTCOME: RULED OUT — new test `real_stack_dir_panel_many_entries_all_have_nonzero_rects_after_load` runs with 120 entries; all 120 children end up with non-zero, in-bounds (x+w ≤ 2.0, y ≥ 0) layout rects. Scale does not change the headless result. / EVIDENCE: `crates/emfileman/src/emDirPanel.rs:1519-1685` (new test).
- HYPOTHESIS C: The nested emDirPanel (created by `emDirEntryPanel::update_content_panel` inside a sub-tree) never reaches `Loaded` because its `PanelCycleEngine` isn't dispatched at deeper nesting levels. F011 fixed dispatch for the first sub-tree level — but emDirPanel → emDirEntryPanel → nested emDirPanel is a deeper chain. / EVIDENCE: not yet gathered — the "Loading NN%" text is painted, which suggests Cycle IS running to update progress; so this hypothesis is weaker.

A directory with thousands of entries takes thousands of frames. C++ runs `TryContinueLoading` in a tight loop per engine tick bounded by time budget (see emDirModel.cpp:116). Porting the time-budget loop is a separate fix (P001 territory), but may also mask the "blank" symptom visibility. / EVIDENCE: emDirPanel.rs:314-370 (Cycle loaded/loading arms) — one call per Cycle.
