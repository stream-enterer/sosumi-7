## Issue

**F010 — Directory listing loads slowly and renders blank after long "Loading NN%" phase**

Prior fix (theme-name validation in `emFileManConfig::Acquire`) changed the symptom from "always black" to "loads eventually, badly". Runtime verification 2026-04-24 (images captured):

1. "Loading NN%" shows for a very long time per directory.
2. After load completes, the listing shows blank (entries not visible).
3. `/root` displays red "Error: Failed to open directory /root: Permission denied (as error 13)" — likely expected for a non-root run, but worth confirming against C++.
4. Some cells show "This file type cannot be shown" red boxes — likely expected `emFilePanel` fallback.

Repro: launch app, navigate into cosmos, zoom into File System or Home.

## Next Steps

1. [ ] Instrument `emDirPanel::Cycle` with logging at entry/exit: which branch taken, loading_done, child_count before/after update_children, entries.len(). Only while reproducing.
2. [ ] Verify at runtime: when a readable directory finishes loading, does `update_children` actually run? Does `ctx.create_child_with` succeed? Is `child_count` > 0 after?
3. [ ] If children are created but not visible: verify `compute_grid_layout` with the ACTUAL panel_height (`ctx.layout_rect().h`) at runtime. Sub-tree nested panel_height may be unexpected.
4. [ ] For the "Loading NN% takes very long" angle: count how many Cycle ticks are consumed per entry. In C++ the loading engine calls `TryContinueLoading` in a tight loop bounded by time budget per tick; Rust does exactly one call per Cycle. If a dir has N entries, that is N frames (~N/60 seconds) — this alone could explain the perceived slowness.
5. [ ] Once evidence locates the break, form a single hypothesis and test it with a minimal fix or regression test.

## Phase 1 — Root Cause Investigation (original)

**E1**: `emFileManThemeData::default()` has `DirContentW: 0.0`, `DirContentH: 0.0`. These stay zero if the theme file never loads. `update_content_panel` lays out the child with `cw=0, ch=0` → invisible.

**E2**: `emFileManViewConfig::Acquire` fell back to `emFileManTheme::Acquire(ctx, "default")` when the config had no theme name. No `default.emFileManTheme` exists → `TryLoad` fails → all dimensions stay 0.0.

**E3**: In C++, `emFileManConfig` constructor (lines 75-79) validates `ThemeName` against `emFileManThemeNames` and replaces it with `GetDefaultThemeName()` (= "Glass1") if invalid. The Rust `emFileManConfig::Acquire` was missing this step entirely.

**E4**: C++ `emFileManViewConfig::emFileManViewConfig` (line 278) simply does `Theme=emFileManTheme::Acquire(GetRootContext(),ThemeName)` — no fallback needed because `emFileManConfig` already guaranteed a valid name.

## Phase 2 — Pattern Analysis

**C++ reference**: `emFileManConfig` constructor acquires `emFileManThemeNames`, checks `IsExistingThemeName`, falls back to `GetDefaultThemeName()`, saves. `emFileManViewConfig` trusts the result — no fallback.

**Rust divergence (resolved by prior fix)**: `emFileManConfig::Acquire` had no theme-name validation. `emFileManViewConfig::Acquire` had an incorrect `"default"` fallback that pointed to a nonexistent theme.

## Phase 3 — Hypotheses

- HYPOTHESIS: Theme-name validation in `emFileManConfig::Acquire` plus removing the `"default"` fallback from `emFileManViewConfig::Acquire` will cause the theme to load with real dimensions so `update_content_panel` produces a non-zero child canvas. / OUTCOME: CONFIRMED (partially — symptom changed from "always black" to "loads then blank") / EVIDENCE: E3, E4; `theme_has_nonzero_dir_content_dimensions` regression test passes.

### Rollback evidence — new runtime failure (2026-04-24)

The theme-name fix removed the "permanently black" symptom but did not fully resolve F010. User-captured screenshots show:
- Outer listing renders entries (red error boxes for files/permission-denied dirs) — so at one level, `update_children` + grid layout works.
- "Loading NN%" is visible for a long time per directory — the progress indicator from `emDirPanel::paint_status` is painting, so `paint_status` is reachable.
- After load completes, the interior is blank — entries either are not created, created with zero-size rects, or created off-screen.
- User hint: "may be related to theme not loading".

### New candidate hypotheses (to be tested, not yet ruled in or out)

- HYPOTHESIS A: Theme parses but some non-tested fields (e.g. `Height`, `NameH`, `PathH`, `InfoH`, `DirPaddingL/T/R/B`, `MinContentVW`) are still zero for the runtime-acquired theme, breaking `compute_grid_layout` or per-entry paint. / OUTCOME: RULED OUT — new test `theme_all_layout_critical_fields_nonzero` asserts 14 layout-critical fields (Height, BackgroundW/H, NameW/H, PathW/H, InfoW/H, DirContentW/H, FileContentW/H, MinContentVW) all load non-zero from Glass1. Test passes. / EVIDENCE: `emFileManViewConfig.rs:663-697`; all 14 fields match values in `res/emFileMan/themes/Glass1.emFileManTheme`.
- HYPOTHESIS B: `emDirPanel::Cycle` reaches `Loaded` but `update_children` either (i) runs with `visible_count == child_count == 0` (model done but entries empty), or (ii) runs and creates children, but `LayoutChildren` sizes them such that they clip off the viewed rect. / EVIDENCE: not yet gathered — needs runtime trace or a targeted integration test simulating the Cycle lifecycle on a real dir.
- HYPOTHESIS C: The nested emDirPanel (created by `emDirEntryPanel::update_content_panel` inside a sub-tree) never reaches `Loaded` because its `PanelCycleEngine` isn't dispatched at deeper nesting levels. F011 fixed dispatch for the first sub-tree level — but emDirPanel → emDirEntryPanel → nested emDirPanel is a deeper chain. / EVIDENCE: not yet gathered — the "Loading NN%" text is painted, which suggests Cycle IS running to update progress; so this hypothesis is weaker.
- HYPOTHESIS D (performance, likely orthogonal to "blank"): `emDirModel::try_continue_loading` advances one name or one entry per call; `emDirPanel::Cycle` calls it at most once per frame. A directory with thousands of entries takes thousands of frames. C++ runs `TryContinueLoading` in a tight loop per engine tick bounded by time budget (see emDirModel.cpp:116). Porting the time-budget loop is a separate fix (P001 territory), but may also mask the "blank" symptom visibility. / EVIDENCE: emDirPanel.rs:314-370 (Cycle loaded/loading arms) — one call per Cycle.

### Priority for next run

Test Hypothesis A first (cheapest, mechanical): extend the theme regression test to assert non-zero values on all layout-critical fields. If any are zero, that is likely the root cause and a fix mirrors the field-read code in `emFileManThemeData::from_rec`.

If A clears, move to B: add an integration test that drives `emDirPanel::Cycle` on a real directory and asserts that after `loading_done == true`, the panel has the expected number of children with non-zero layout rects.
