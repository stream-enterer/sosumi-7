## Phase 1 — Root Cause Investigation (original)

**E1**: `emFileManThemeData::default()` has `DirContentW: 0.0`, `DirContentH: 0.0`. These stay zero if the theme file never loads. `update_content_panel` lays out the child with `cw=0, ch=0` → invisible.

**E2**: `emFileManViewConfig::Acquire` fell back to `emFileManTheme::Acquire(ctx, "default")` when the config had no theme name. No `default.emFileManTheme` exists → `TryLoad` fails → all dimensions stay 0.0.

**E3**: In C++, `emFileManConfig` constructor (lines 75-79) validates `ThemeName` against `emFileManThemeNames` and replaces it with `GetDefaultThemeName()` (= "Glass1") if invalid. The Rust `emFileManConfig::Acquire` was missing this step entirely.

**E4**: C++ `emFileManViewConfig::emFileManViewConfig` (line 278) simply does `Theme=emFileManTheme::Acquire(GetRootContext(),ThemeName)` — no fallback needed because `emFileManConfig` already guaranteed a valid name.

## Phase 2 — Pattern Analysis

**C++ reference**: `emFileManConfig` constructor acquires `emFileManThemeNames`, checks `IsExistingThemeName`, falls back to `GetDefaultThemeName()`, saves. `emFileManViewConfig` trusts the result — no fallback.

**Rust divergence (resolved by prior fix)**: `emFileManConfig::Acquire` had no theme-name validation. `emFileManViewConfig::Acquire` had an incorrect `"default"` fallback that pointed to a nonexistent theme.

### Phase 3 Hypotheses — updated 2026-04-24 pm

- **NEW HYPOTHESIS E (root candidate):** *The inner sub-view's `emView::VisitPanel` / `VisitFullsized` does not actually drive the view's animator on this Rust port.* StartupEngine's `VisitFullsized(":")` returns without effect; programmatic `VisitPanel` from the control channel returns without effect; `wait_idle` reports idle anyway because no animator is queued.
