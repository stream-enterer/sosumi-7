## Issue

**F012 — Control panel ContentControlPanel never created — context-sensitive app icons missing**

In C++ Eagle Mode, emMainControlPanel has two children: lMain (general controls + bookmarks,
weight 11.37) and ContentControlPanel (context-specific icons, weight 21.32). ContentControlPanel
is created via ContentView.CreateControlPanel() and changes based on the active panel in the
content view. In the Rust port, emMainControlPanel::create_children() never creates the
ContentControlPanel child — emDirEntryPanel::CreateControlPanel (which returns emFileManControlPanel)
is never called. ControlPanelBridge only logs.

**Repro:** Launch app. Zoom into the file system panel. The right portion of the control panel
(where emFileManControlPanel would appear: Select Files, Help, etc.) is absent. In C++ Eagle Mode,
the right half of the control panel changes context-sensitively based on what is zoomed into.

## Phase 1 — Root Cause Investigation

**E1.1** (`emMainWindow.rs:1039`): `win.view_mut().set_control_panel_signal(cp_signal)` sets the
signal on the OUTER window's `emView`, not on the content sub-view's `sub_view`. The content
sub-view's `sub_view.control_panel_signal` is `None` — it never fires when the active panel changes.

**E1.2** (`emMainWindow.rs:1047-1051`): ControlPanelBridge has `_ctrl_view_id: root_id` and
`_content_view_id: root_id` — both set to the outer tree's root panel ID (wrong). They should be
`ctrl_id` and `content_id` (the ctrl and content sub-view panel IDs in the outer tree).

**E1.3** (`emMainWindow.rs:807-812`): `ControlPanelBridge::Cycle` is a log stub — never creates
ContentControlPanel.

**E1.4** (`emMainControlPanel.rs:187-211`): `create_children()` creates `lMain` but never creates
a ContentControlPanel child. `_content_control_panel: Option<PanelId>` is always None.

**E1.5** (scheduler dispatch, `emScheduler.rs:638-655` + `emPanelCycleEngine.rs:115-183`):
For SubView-scope panel engines, the scheduler takes the outer tree (`win.take_tree()`) and passes
it as `ctx.tree`. The PanelCycleEngine constructs `ectx` with `tree: None` and `windows: ctx.windows`.
This means emMainControlPanel::Cycle cannot call `win.take_tree()` — the tree is already taken.
Cross-tree access from a SubView panel's Cycle is not possible. A Framework-scope engine
(ControlPanelBridge) is the correct cross-tree vehicle.

**Root cause chain:**
1. User sees no context-sensitive icons in control panel
2. ← ContentControlPanel never created in ctrl sub-tree
3. ← ControlPanelBridge::Cycle is a log stub, never creates ContentControlPanel (E1.3)
4. ← ControlPanelBridge never fires: `cp_signal` is set on outer window's emView, not content sub-view (E1.1)
5. ← Content sub-view's `sub_view.control_panel_signal` is None (E1.1)
6. (Secondary): ControlPanelBridge uses wrong panel IDs (`root_id` instead of `ctrl_id`/`content_id`) (E1.2)

## Phase 2 — Pattern Analysis

**C++ behavior** (`emMainControlPanel.cpp`):
- Constructor receives `emView & contentView` (the content sub-view's emView)
- `AddWakeUpSignal(ContentView.GetControlPanelSignal())` subscribes to content view's intrinsic signal
- `RecreateContentControlPanel()` called at construction AND in Cycle whenever signal fires
- `RecreateContentControlPanel()`: `ContentView.CreateControlPanel(*this, "context")` — walks active
  panel chain in content view, calls CreateControlPanel on each, creates child under emMainControlPanel

**C++ does NOT have a bridge engine.** emMainControlPanel IS the entity that handles signal + recreation.

**Rust difference (language-forced):**
- emMainControlPanel is a SubView-scope PanelBehavior; its Cycle cannot access the outer tree
  (scheduler takes the outer tree before dispatching SubView engines — see E1.5)
- ControlPanelBridge (Framework-scope) IS the correct pattern: Framework engines have `ctx.windows`
  with the outer tree available via `win.take_tree()`
- But ControlPanelBridge is currently: (a) wired to wrong emView signal, (b) has wrong panel IDs,
  (c) is a stub

**Available infrastructure (confirmed working):**
- `PanelTree::create_control_panel_in(content_tree, active_id, ctrl_tree, ctrl_root, "context", tallness)` — cross-tree creation
- `emSubViewPanel::sub_tree_mut()` — ctrl/content sub-tree access
- `emSubViewPanel::sub_view_mut()` — sub_view with `set_control_panel_signal()`
- `emView::GetActivePanel() -> Option<PanelId>` — active panel in content sub-view
- `PanelTree::remove(id, sched)` — removes old ContentControlPanel

## Phase 3 — Hypotheses

HYPOTHESIS: Fix the three ControlPanelBridge bugs (signal wired to content sub-view instead of outer view; correct ctrl/content panel IDs; implement Cycle with create_control_panel_in) and add set_content_control_panel() to emMainControlPanel — ContentControlPanel will appear and update context-sensitively.
OUTCOME: CONFIRMED (implemented in Phase 4 and verified via cargo check + clippy + nextest)
EVIDENCE: E1.1, E1.2, E1.3 (root cause); E1.4, E1.5 (forced divergence); Phase 2 (infrastructure available)

## Root Cause

Three ControlPanelBridge bugs: (1) `cp_signal` set on outer window's emView instead of content
sub-view's `sub_view` — signal never fires on active panel change; (2) both panel IDs use `root_id`
instead of `ctrl_id`/`content_id`; (3) `Cycle` is a log stub. These combine to prevent
ContentControlPanel from ever being created. The bridge pattern itself is DIVERGED (language-forced):
C++ emMainControlPanel directly subscribes to ContentView.GetControlPanelSignal() and calls
CreateControlPanel from its own Cycle, but Rust SubView-scope engines cannot access the outer tree
while the scheduler holds it.

**Fix:**
1. Wire `cp_signal` on `content_svp.sub_view_mut()` (not outer window's emView)
2. Fix ControlPanelBridge: add `window_id`, fix panel ID fields to `ctrl_id`/`content_id`
3. Implement ControlPanelBridge::Cycle: take outer tree → take both SVP behaviors → delete old CCP
   → `create_control_panel_in` → `set_content_control_panel` on emMainControlPanel → put back
4. Add `set_content_control_panel(id)` to emMainControlPanel (sets layout weight 21.32)
