- [ ] **Step 3: Determine when slots can be `None` in practice**

After resize, the cycle restarts.

### Task 9: I.2 — Tile backing-store init color is not observable

- [ ] **Step 1: Confirm tile init color**

- [ ] **Step 2: Cross-reference with O.5 finding**

A `None` compositor slot is solid black via load-clear (per O.5). A `Some` slot whose tile content has any pixel with alpha < 255 will composite RGB onto the load-clear black. The init color RGBA `(0,0,0,0)` means a tile that is allocated but never painted (if such a state can happen) blends the load-clear black through fully — visible black.

- [ ] **Step 3: Determine if "allocated but never painted" can happen**

A tile is allocated by `Tile::new()` inside `TileCache::get_or_create` (`emViewRendererTileCache.rs:70-78`), called from `emWindow::render` during the dirty-counting loop (`emWindow.rs:622`). At creation, `dirty: true`. The render strategies then paint dirty tiles. So a freshly-allocated tile is dirty; the same render cycle paints it.

BUT — paint may not fully cover the tile. Rules I.1 + I.3 + III interact: if `view.Paint`'s conditional clear is skipped (opaque-and-covering SVP), and the panel doesn't paint the entire tile region (e.g., tile sits at the edge of the panel's clip rect), the un-painted pixels remain at the init color RGBA `(0,0,0,0)`.
