### Drift-prevention notes (carry-forward into Task-dispatch prompts)

1. Explicit grep assertion at end of each task (e.g., "confirm `rg -w 'sub_scheduler' crates/` returns zero before marking complete").
2. Prohibition on relocating the field to any other struct. The goal is deletion, not motion.
3. Pre-commit `--no-verify` usage permitted ONLY for checkpoint commits within a single task; final task commit must pass full gate.

The user's standing rule is in CLAUDE.md: "The entire goal of this plan is to CLEAN UP divergences, workarounds, and hacks; not to create new ones." All task prompts should echo this.

The test asserts the recorded pointer equals a pre-dispatch snapshot of the innermost `sub_tree_mut` pointer, proving the two-hop take/put walk reaches the correct tree. Passes.

Test-only concession: the depth-2 test leaks the nested `emSubViewPanel` behaviors via `std::mem::forget` at teardown. Reason: `emSubViewPanel::new` constructs its own `sub_scheduler` with registered `UpdateEngineClass`/`VisitingVAEngineClass` engines; properly tearing them down mid-test requires the full `emSubViewPanel` teardown helper (which lives inside `emSubViewPanel.rs` tests and depends on `sub_view.update_engine_id`). Borrow-checker: the dispatch walk is a small recursive function over `&TreeLocation`, not an explicit stack. The `take_behavior`/`put_behavior` pairing is naturally lexically scoped by each recursive call — no issue with nested borrows. Cloning the `TreeLocation` (shallow `PanelId` + `Box<TreeLocation>` chain) out of `engine_locations` before the walk avoids aliasing the scheduler's inner state during dispatch; clone cost is one `Box::new` per `SubView` hop, which matches the rare-path cost profile.

An `emSubViewPanel::sub_tree` calls `set_tree_location(SubView(outer_panel_id, Outer))` in `emSubViewPanel::new`, so every subsequent `create_child` in the sub-tree tags its adapter engine for outer-scheduler dispatch through the sub-view. - `crates/emmain/src/emMainWindow.rs` — call-site restructure for `ctrl_svp` and `content_svp`: `create_child` runs FIRST (outer slot id obtained), then a `SchedCtx` is built around `app.scheduler`, then `emSubViewPanel::new(ctx.clone(), outer_id, &mut sc)` runs with the id in hand. This matches the "caller must `create_child` the outer slot first" pattern from the Task-3 plan text verbatim.

Call-site restructuring pattern (summary): in every `emSubViewPanel::new` call site — both production (`emMainWindow::create_main_window` for ctrl/content svp) and tests (`SvpTestHarness` in `emSubViewPanel.rs` sp8 tests, the Task-2 depth-2 test in `emScheduler.rs`) — the sequence is:

1. Create the outer-tree parent/root so a valid `tree` exists.
2. `create_child` the outer slot that will hold the sub-view — capture the returned `outer_panel_id`.
3. Build a `SchedCtx` wrapping the outer scheduler.
4. `emSubViewPanel::new(parent_context, outer_panel_id, &mut sc)` — this registers all sub-view-owned engines on the outer scheduler with `SubView(outer_panel_id, Outer)` during construction.
5. `tree.set_behavior(outer_panel_id, Box::new(svp))` — install.

No two-phase init or `register_engines` post-construction helper was needed; step 2 before step 4 resolves the cyclic dependency the plan warned about. The `set_behavior` in step 5 happens AFTER `new` returns, so the dispatch walk from the outer scheduler can't fire against this sub-view until it's installed — but `new` already populated `engine_locations[outer_panel_id] = SubView(...)` for the sub-view's engines. If the outer scheduler fires one of those engines before step 5 installs the behavior, `dispatch_with_resolved_tree` would panic on the missing behavior. In practice this never happens: there is no `DoTimeSlice` call between steps 4 and 5 in any call site (both are synchronous in the same code block).
