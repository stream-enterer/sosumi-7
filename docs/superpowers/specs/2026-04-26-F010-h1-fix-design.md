# F010 H1 fix design — `DrawOp::Clear` variant

## Goal

Fix the `emPainter::Clear()` recording-mode dispatch hole confirmed by F010 investigation entry `log/0007-h1-experiment.md`. After this fix, calls to `painter.Clear(color)` against a recording (`DrawList`) painter target record an op that, at replay, fills the painter's current clip with the given color — matching the direct-mode behavior.

## Background

`emPainter::Clear` currently uses `require_direct()`, which returns `None` for `DrawList` targets. The function early-returns silently in recording mode. All seven panel callers (`emDirPanel`, `emStarFieldPanel`, `emFileLinkPanel` ×2, `emDirStatPanel`, `emMainPanel`, `emStocksFilePanel`) have therefore been silently failing whenever paint runs through the parallel tile compositor's recording path.

The investigation's pre-registration table is at `docs/debug/investigations/F010-investigation/hypotheses/`. The forbidden-fix-shapes checklist at `docs/debug/investigations/F010-investigation/forbidden-fix-shapes.md` rules out: feature flags, dispatch reroutes that bypass the broken path, and panel-level workarounds. The permitted shape — *"add a `DrawOp::Clear { color }` variant + record path + replay handler at the painter layer (fixes the broken path itself)"* — is what this spec implements.

## Architecture

The fix touches two production files:

1. **`crates/emcore/src/emPainterDrawList.rs`** — adds a `Clear { color }` struct variant to the `DrawOp` enum (positioned between the state ops and the shape ops, since Clear is conceptually the simplest paint op), and one match arm in the replay loop that delegates to the public method.

2. **`crates/emcore/src/emPainter.rs`** — replaces the `require_direct()` call inside `Clear` with `try_record(DrawOp::Clear { color })`. Direct-mode behavior is unchanged — same `proof` token, same `fill_rect_pixels` call.

`ClearWithCanvas` is unchanged. It already records correctly because it delegates to `PaintRect`.

### Replay handler design (Style A)

```rust
DrawOp::Clear { color } => painter.Clear(*color),
```

Mirrors the existing pattern used by every other shape variant (e.g. `DrawOp::PaintRect { … } => painter.PaintRect(…)`, `DrawOp::PushState => painter.push_state()`). At replay time the receiving painter is always direct-mode (a per-tile `emImage` target), so this re-entry into `Clear` runs `try_record` against the direct target, gets `Some(DirectProof)`, and falls through to the pixel fill. Cost: one extra `try_record` call per Clear at replay (a `PaintTarget` match plus a `state_snapshot`); negligible.

### State capture

`try_record` captures the painter's `state_snapshot` (clip, transform, alpha, canvas color) into the `RecordedOp` automatically. The replay handler doesn't need to read clip dimensions from the captured state — at replay time, the painter's running state is reconstructed by the preceding `PushState`/`PopState`/`ClipRect`/`SetTransformation` ops, so when `painter.Clear(*color)` runs, `self.state.clip` already holds the correct values.

### Send/Sync

`DrawOp::Clear { color: emColor }` uses only `Copy` types. No new `unsafe impl` is required — the existing variant-wide `Send`/`Sync` proofs (which already cover variants holding `*const emImage`, etc.) cover this trivially.

## Components

### `DrawOp::Clear` variant

Positioned in the enum immediately after `SetAlpha(u8)` and before `PaintRect`. Struct variant, single field:

```rust
Clear {
    color: emColor,
},
```

### Replay match arm

In the big match in `emPainterDrawList::replay` (around line 446), after the state-management arms and before `PaintRect`:

```rust
DrawOp::Clear { color } => painter.Clear(*color),
```

### `emPainter::Clear` body

```rust
pub fn Clear(&mut self, color: emColor) {
    let Some(proof) = self.try_record(DrawOp::Clear { color }) else {
        return;
    };
    let x = self.state.clip.x1 as i32;
    let y = self.state.clip.y1 as i32;
    let w = self.state.clip.x2.ceil() as i32 - x;
    let h = self.state.clip.y2.ceil() as i32 - y;
    self.fill_rect_pixels(proof, x, y, w, h, color);
}
```

The diff vs. current is a single token change in line 1: `require_direct()` → `try_record(DrawOp::Clear { color })`.

## Testing strategy

### Test 1 — Inversion of the H1 falsification test

`crates/emcore/tests/f010_h1_clear_recording.rs` currently asserts `ops_added_by_clear == 0` (passes when H1 holds). After this fix it must assert `ops_added_by_clear == 1` and that the recorded op is the `DrawOp::Clear` variant carrying the same color the caller passed. Renaming the test function to `f010_h1_clear_records_one_op` clarifies the inversion.

The test continues to write the same JSON artifact at `docs/debug/investigations/F010-investigation/artifacts/H1-ops.json` — its content shifts to reflect the post-fix observation. A `revise` log entry under `docs/debug/investigations/F010-investigation/log/` will be added separately (out of this spec's scope) to record the inversion.

### Test 2 — Round-trip pixel correctness

New file `crates/emcore/tests/f010_h1_clear_replay_pixels.rs`. Drives the full record→replay cycle:

1. Construct a recording painter, set a known clip (e.g. 50,40,750,540), call `Clear(rgba(200, 100, 50, 255))`. Verify the captured `RecordedOp` has the expected variant, color, and clip.
2. Construct a direct painter backed by an `emImage` of matching size, set the same clip via `SetClipping`, replay the captured op via the existing `replay` entry point. Verify pixels inside the clip rect match the requested color (sample center pixel + corners) and pixels outside the clip are unchanged from a known sentinel pre-fill (e.g. RGBA(0,0,0,0)).

This test does the falsification of "Clear records but doesn't actually paint at replay" — closes the loop on the fix.

### Test 3 — All seven panel callers

No new tests. The existing panel paint tests in `crates/emfileman/tests/`, `crates/emstocks/tests/`, etc. exercise these callers. After the fix, any test that recorded zero panel-interior fills before should now record one. Pre-existing golden tests (under `crates/eaglemode/tests/golden/`) act as the regression net for op-stream parity vs. C++.

### Golden test impact

If C++ goldens were captured against C++ paint output (which uses direct-mode Clear), they already contain the expected post-Clear pixels. Rust output before this fix was missing those pixels (showing whatever was in the tile pre-fill underneath). After this fix Rust matches C++. Expected golden movement: **toward parity, not away**. If a golden moves in the other direction, it's diagnostic — likely an unrelated divergence the fix exposes.

The pre-commit hook runs goldens automatically. Movement that improves divergence counts (per `scripts/divergence_report.py --diff`) is acceptable; movement that worsens them gates the commit and gets investigated case-by-case.

## Risk and rollback

Risk: low. The change is purely additive in the enum and its replay loop, and a single-line change in `Clear`. Direct-mode behavior is byte-identical (same proof, same fill).

Possible failure modes:
1. **Visible F010 symptom doesn't fully clear up** — this would mean H1 is necessary but not sufficient. The investigation's resume protocol at `docs/debug/investigations/F010-investigation/RESUME-INVESTIGATION.md` covers this; the locked hypothesis table is ready for Tasks 3.2-3.8 to resume.
2. **A non-F010 caller relied on the silent no-op** — extremely unlikely (all seven callers want their backgrounds visible), but would surface as an unexpected paint regression in another panel. Mitigated by golden-test coverage.
3. **Golden test moves negatively** — investigated per `scripts/diff_draw_ops.py` to identify the divergence shape.

Rollback: `git revert <commit>` cleanly reverses the change. The investigation evidence and resume protocol are independent of the fix commit and survive a revert.

## Branch and commit strategy

Work continues on `f010-investigation` (the branch the investigation was conducted on). The fix lands as one or more commits here, then the branch eventually merges to `main` carrying both the investigation log and the fix together.

A single commit is preferred (one logical change: add `DrawOp::Clear` variant + record path + replay handler + tests). The `revise` log entry on the investigation log is a separate commit so the methodology log and the production code change have independent SHAs.

## Out of scope

- Other panel paint correctness (the `revise` entries / Phase 4 verification land separately).
- Updating ISSUES.json — that's a closeout step after manual GUI verification (Task 4.2 of the investigation methodology).
- Refactoring `require_direct()` itself or removing other callers of it (other call sites have legitimate "no DrawOp variant possible" reasons, e.g., pixel readback).
- Changing `ClearWithCanvas` behavior. It already records correctly via `PaintRect`.
