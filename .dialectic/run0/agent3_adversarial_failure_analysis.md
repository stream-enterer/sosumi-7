# Adversarial Failure Analysis: Approach A vs Approach B

This document catalogs failure modes of two approaches to closing the 37 remaining golden test failures in eaglemode-rs. It does not recommend either approach. It diagnoses how each can fail, stall, or produce false confidence, with emphasis on compounding and non-obvious mechanisms.

---

## Part I: Failure Modes of Approach A (Continue Current)

### 1. Hypothesis Cascade Exhaustion and False Elimination

Three hypotheses for Group A+B have been investigated and "disproven." Each elimination rests on the claim that a specific subsystem's outputs match between C++ and Rust. But what does "match" mean in context?

**The precision trap.** Investigation 2 concluded "9-slice transform parameters match exactly." But the comparison was between Rust's `paint_9slice_section` parameter computation and C++ `ScanlineTool::Init`'s parameter computation. These are not the same function -- they are two different functions that are supposed to produce equivalent results. The comparison checked the *inputs* to the divergent function, not the function's internal state. If the C++ `ScanlineTool::Init` computes `ImgMap`, `ImgDX`, `ImgDY`, `ImgSX`, `ImgSY`, `TX`, `TY`, `TDX`, `TDY`, `ODX`, `ODY` (12 derived values from those inputs), and the Rust `paint_9slice_section` computes its own derived values through a different code path, then "inputs match" does not imply "derived state matches." The investigation eliminated the hypothesis at the wrong level of abstraction.

**Compounding mechanism:** Each false elimination narrows the search space *incorrectly*. After three eliminations, the investigator believes: "It's not the area sampling loop, not the transform parameters, not the color mapping." This exclusion set becomes the starting assumption for investigation 4. But if investigation 2's elimination was wrong (the transforms do differ, just not at the level that was checked), investigation 4 will not revisit it. The excluded region of the search space becomes a permanent blind spot. After N rounds, the blind spots are not additive -- they are multiplicative, because each new investigation's search direction is constrained by all previous exclusions.

**The "disproven incorrectly" scenario.** Investigation 3 concluded "PaintImageColored is not called at divergent pixels." This was verified by checking which code path handles specific pixel coordinates. But if the divergence is in a shared upstream function (e.g., sub-pixel edge calculation, clip rectangle rounding) that affects *where* pixels land rather than *how* they're colored, then the divergent pixels in the C++ output simply don't exist at the same coordinates in the Rust output. The investigation compared pixel values at fixed coordinates, but the coordinates themselves may be the divergence. This class of error -- comparing the wrong thing correctly -- is invisible to the investigator because the comparison itself produces clean, definitive-looking results.

### 2. Fog Compounding Through Partial Observations

The "structurally equivalent with minor differences" verdict is the most dangerous output an agent can produce. It sounds rigorous. It implies thorough comparison. But it is structurally unfalsifiable within the investigation's scope.

**What "minor differences" actually means.** When an agent reads `paint_9slice_section` (100 lines of Rust) and `ScanlineTool::Init` (280 lines of C++ with macro-expanded function pointer tables, goto statements, pointer arithmetic on `ImgMap`, stride recalculation for downscaling), the agent is comparing two code artifacts of different size, different structure, and different idiom. The agent's comparison is necessarily high-level: "both compute transform parameters from source/dest rectangles." But the C++ code does this:

```cpp
ImgMap=texture.GetImage().GetMap()+(sy*(size_t)iw+sx)*channels;
ImgDX=channels;
ImgDY=((ssize_t)iw)*channels;
```

This sets `ImgMap` to point directly into the image buffer with byte-level pointer arithmetic. The Rust code doesn't use raw pointers this way. It passes section bounds as `SectionBounds { ox, oy, w, h }` to the interpolation functions. Whether these two representations produce identical sampling behavior depends on how every downstream consumer interprets them. The agent cannot verify this without tracing every consumer -- and the C++ consumers are macro-expanded templates that the agent never sees in their final form.

**The CONCAT macro invisibility problem.** The C++ interpolation functions are generated by `#include`-based template expansion. `emPainter_ScTlIntImg.cpp` defines `InterpolateImageAreaSampled` inside a `#define`/`#include` cycle that is re-included with different `CHANNELS` and `EXTENSION` settings. The actual function that executes for a 4-channel clamped area-sampled image is `InterpolateImageAreaSampledEtCs4` -- a name constructed by macro concatenation. The agent sees the template function with `CHANNELS` as a compile-time parameter, but the behavioral differences between channel counts (especially the alpha channel handling when `CHANNELS==4`) are embedded in `#if CHANNELS>=4` guards within the template. An agent reading the template sees the general case. The specific case -- which is the one that actually runs for the failing tests -- requires mentally expanding the macros, which agents do unreliably.

**Compounding mechanism:** Each round of fog adds a layer of "we checked this and it was fine" to the project's institutional memory. Future investigations inherit this fog as settled fact. But the fog is not evenly distributed -- it concentrates around the exact areas where the divergence lives, because those are the areas that have been investigated most heavily and declared equivalent.

### 3. Cleanup-as-Progress Illusion

The pattern is concrete and documented: Investigation round 1 targeted Group A+B but fixed `blend_colored_scanline` (5 tests in other groups). Investigation round 2 targeted Group A+B but found that HowTo text wiring was missing (20 tests). Each round produced real value. Each round moved the overall pass count upward. Each round failed to fix a single Group A+B test.

**The systematic diversion mechanism.** When an agent investigates Group A+B, it reads the rendering pipeline end-to-end. Along the way, it encounters genuine bugs in adjacent code. These bugs are real, visible, and fixable. The agent fixes them because (a) they are in scope of the investigation, (b) fixing them feels productive, and (c) leaving known bugs unfixed feels irresponsible. But each side-quest fix consumes context window space, investigation time, and decision-making bandwidth. The agent returns to Group A+B with less remaining capacity.

**The deeper trap.** The side-quest fixes make the codebase healthier. They reduce noise in the divergence log. They make future comparisons cleaner. All of this is genuinely valuable. But it creates a reward signal that reinforces the behavior of broad investigation over deep investigation. The agent learns (in the reinforcement-learning sense of "learns" from its context window) that investigating Group A+B broadly produces fixes, while investigating Group A+B deeply produces null results. This biases subsequent investigations toward breadth, which is exactly the wrong direction for a root cause that lives in a specific 10-line section of a specific function.

**Compounding mechanism:** After K rounds, the project has K successful side-quest fixes and zero Group A+B fixes. The ratio of effort-to-progress for Group A+B approaches infinity. But the overall project metrics (204/241 at start, now at some higher number) look healthy. There is no metric that tracks "effort specifically allocated to Group A+B that produced no result." The failure is invisible in the aggregate.

### 4. Golden Test Tunnel Vision and Backward-Trace Collapse

A golden test failure says: "pixel (32, 289): actual=rgb(123,132,162) expected=rgb(119,129,159)." The maximum channel difference is 4. This tells the investigator that some rendering operation at or near this pixel produced a result that is off by a small amount across all three channels.

**The backward trace problem.** To find the cause, the investigator must trace backward: What drew this pixel? Was it a border image section? Which section (of the 9)? What interpolation method? What source pixel(s) did it sample? What weights were applied? What blending formula combined it with the background? Each step requires reading both the C++ and Rust implementations of that step, comparing them, and determining whether they match.

The C++ interpolation for area sampling (`InterpolateImageAreaSampled` in `emPainter_ScTlIntImg.cpp`) is ~230 lines of dense integer arithmetic with carry state, loop unrolling, and pointer arithmetic. The Rust equivalent (`interpolate_scanline_area_sampled` in `emPainterInterpolation.rs`) is a separate implementation. Comparing them requires holding both in context simultaneously. For an agent with a context window, this means ~460 lines of comparison code plus the surrounding context (function signatures, struct definitions, constant values). This is feasible for a single function.

But the backward trace doesn't stop at one function. The area sampling result feeds into `PaintScanlineInt` (the blending function), which is itself macro-expanded across pixel formats and canvas-visibility configurations. The C++ version (`PaintScanlineIntCs4Ps4Cv` for 4-channel, 4-byte pixel format, canvas-visible) is constructed by macro concatenation and never appears literally in any source file. The agent must mentally expand the template to understand it. This expansion is unreliable.

**Compounding mechanism:** Each backward-trace step has a probability P of the agent making an error (misreading a macro expansion, confusing a variable's value at two different points in a loop, misunderstanding integer promotion rules). If the trace is 5 steps deep and P=0.1 per step, the probability of an error-free trace is 0.9^5 = 0.59. For a 10-step trace (which is realistic for "pixel output -> blend -> interpolation -> transform -> section bounds -> clip -> sub-pixel edge -> source coordinate"), the probability drops to 0.9^10 = 0.35. More than half of deep traces will contain at least one error.

### 5. Agent Confirmation Bias in Code Comparison

When an agent reads 200 lines of C++ and 200 lines of Rust and must determine equivalence, it has systematic blind spots.

**Integer promotion and widening.** C++ `int * int` produces an `int` (with potential overflow). Rust `i32 * i32` also produces `i32` (with panic in debug mode, wrap in release). But C++ `unsigned * signed` involves implicit conversion to unsigned, while Rust requires explicit casts. If the C++ code computes `(unsigned)a * (int)b` where `b` is negative, the result is well-defined but surprising in C++. The Rust port might cast differently. Agents rarely track signedness through multi-step arithmetic.

**Evaluation order in compound expressions.** C++ does not guarantee evaluation order of function arguments (pre-C++17). The C++ painter code uses function-pointer dispatch: `sct.PaintScanline(sct, x, y, w, opacityBeg, opacity, opacityEnd)`. If any of these arguments has side effects or depends on mutable state that other arguments also read, evaluation order matters. In Rust, function arguments are evaluated left-to-right. An agent comparing the two will see "same function, same arguments" and conclude equivalence, missing the evaluation-order divergence.

**The scanline carry-state problem.** The C++ area sampling loop carries state between pixels within a scanline (`AreaSampleCarryState`). The Rust implementation has this too. But carry state means that a 1-LSB error in pixel N propagates to pixel N+1. An agent comparing the loop body for a single iteration will see matching code. The divergence only manifests after accumulating across many pixels in a row. This is a class of bug that is invisible to static code comparison and only visible through runtime tracing.

**Compounding mechanism:** Each "structurally equivalent" verdict increases confidence in the Rust implementation's correctness. But the verdicts are produced by a process (agent code comparison) that has known blind spots. The confidence is not calibrated to the process's actual reliability. After several "structurally equivalent" verdicts, the assumption becomes "the code is correct, the problem must be elsewhere" -- which forecloses investigation of the code that was already reviewed.

### 6. Sunk Cost Trajectory Lock

Three investigation rounds have targeted Group A+B (21 tests, the largest group). The implicit reasoning is: "fix the largest group first for maximum impact." But this reasoning assumes all groups are equally hard. They are not.

**The hardness distribution.** Group A+B involves 9-slice border image rendering, which exercises: (a) 9-slice subdivision, (b) section-level transform computation, (c) downscale pre-reduction with stride modification, (d) area-sampled interpolation with carry state, (e) compositing with canvas color awareness. This is the deepest code path in the renderer. Meanwhile, G2 (polygon rasterizer, 2 tests) may be a single sign error. G7 (gradient, 2 tests) may be a single coefficient. G9 (polyline stroke, 1 test) may be an endpoint convention difference. These smaller groups could each fall to a single targeted investigation.

**The opportunity cost.** Each round spent on Group A+B that produces zero fixes is a round not spent on a group that might have fallen in one round. If G2+G3+G7+G9 (7 tests) each take one round, that's 4 rounds and 7 tests fixed. If Group A+B takes 3 more rounds and then finally falls, that's 3 rounds and 21 tests fixed. The expected value depends on the probability of each round succeeding, which is unknown. But the trajectory lock means this calculation is never made -- the next round defaults to "continue on A+B" because that's where the investigation history lives.

**Compounding mechanism:** Each additional round on A+B increases the sunk cost, which increases the psychological pressure to continue, which decreases the probability of switching to a higher-expected-value target. This is a classic sunk cost escalation pattern, and it operates even on agents because the context window contains the investigation history, which primes the agent to continue the current investigation rather than pivot.

## Part II: Failure Modes of Approach B (Incremental Rebuild)

### 7. Intermediate Equivalence Does Not Compose

Approach B proposes comparing C++ and Rust at stage boundaries. Suppose we verify: (a) 9-slice subdivision produces identical section rectangles, (b) transform computation produces identical `TDX`, `TDY`, `TX`, `TY` values, (c) interpolation produces identical pixel values for those transform parameters. If all three pass, the composition should match. But it may not.

**The state-coupling problem.** C++ `ScanlineTool::Init` computes all derived state in a single function. It modifies `ImgMap`, `ImgDX`, `ImgDY` (pointer arithmetic for stride reduction), then uses those modified values to compute `TDX` and `TDY`. The Rust code computes stride reduction and transform parameters in separate functions. If the intermediate harness tests transform computation with *original* `ImgW`/`ImgH` values (before stride reduction), the transforms will match. But in the actual pipeline, stride reduction modifies `ImgW`/`ImgH` before transform computation. The intermediate test passes; the end-to-end test fails.

This is not hypothetical. The C++ code at line 314-337 of `emPainter_ScTl.cpp` modifies `ImgW`, `ImgDX`, `ImgSX`, `ImgMap` in-place, then uses the *modified* `ImgW` to recompute `tdx` at line 323: `tdx=ImgW*((emInt64)1<<24)/tw`. If the Rust code computes `tdx` before or after the equivalent of stride reduction, and the intermediate harness tests them separately, the harness cannot catch the ordering dependency.

**Compounding mechanism:** Each stage that passes its intermediate test adds a "proven correct" label. When the composition fails, the investigator must now re-investigate every "proven correct" stage to find the composition-dependent failure. But the "proven correct" labels create cognitive resistance to re-investigation. The intermediate tests become a source of false confidence that is harder to overcome than having no tests at all, because "we already proved that stage works."

### 8. Decomposition Mismatch Between C++ and Rust Pipelines

The fundamental assumption of Approach B is that both pipelines decompose into the same stages. They do not.

**C++ pipeline.** `PaintImage` validates the texture, calls `ScanlineTool::Init` (which dispatches through macro-expanded function pointer tables), then enters the scanline loop. `ScanlineTool::Init` is a 280-line monolithic function that handles COLOR, IMAGE, IMAGE_COLORED, LINEAR_GRADIENT, and RADIAL_GRADIENT textures in a single switch statement with gotos. For image textures, it computes pointer offsets, stride reduction, and transform parameters all in one flow.

**Rust pipeline.** `paint_9slice_section` is a standalone function that computes transforms, calls `area_sample_transform_24`, sets up `SectionBounds`, then enters a scanline loop that calls `interpolate_scanline_area_sampled` and `blend_scanline`. There is no equivalent of `ScanlineTool::Init` -- its responsibilities are distributed across `paint_9slice_section`, `area_sample_transform_24`, `InterpolationBuffer`, and `BlendMode`.

**Where to compare?** There is no clean "stage boundary" that exists in both pipelines. C++ `ScanlineTool::Init` output state (12+ fields) maps to Rust's `Transform24`, `SectionBounds`, `InterpolationBuffer`, and `BlendMode` -- four separate structs. To compare "stage 2 output," you must first define a mapping between these representations. That mapping itself embeds assumptions about what's equivalent. If the mapping is wrong, the comparison passes but the pipelines still diverge.

**Compounding mechanism:** The decomposition choice becomes a commitment. Once you've built harnesses for a particular decomposition, changing the decomposition means rebuilding the harnesses. This creates inertia around the initial decomposition choice, even if it proves to be the wrong one. The infrastructure investment locks in architectural assumptions.

### 9. Infrastructure Cost and the Instrumentation-Introduces-Bugs Problem

Building intermediate comparison harnesses requires instrumenting the C++ code to emit intermediate values. The C++ code uses macro-expanded templates, `#include`-based code generation, and raw pointer arithmetic. Instrumenting it is non-trivial.

**The C++ instrumentation problem.** To emit `ScanlineTool` state after `Init`, you must add print statements to `emPainter_ScTl.cpp`. But this file is included by the build system in a specific way, and modifying it changes the binary. The golden reference images were generated by the unmodified binary. If the instrumentation changes any behavior (timing, memory layout, alignment), the reference images are invalid for the instrumented binary.

More subtly: the C++ code stores `ImgMap` as a raw pointer. To emit its value for comparison, you need to emit the pointer offset from the image base. But `ImgMap` is modified during stride reduction (line 320: `ImgMap+=ImgDX*(t>>1)`). Emitting it before vs. after stride reduction gives different values. You must instrument at exactly the right point, which requires understanding the code flow -- the same understanding that the investigation is trying to verify.

**The Rust instrumentation problem.** The Rust code is more amenable to instrumentation, but adding debug output to hot loops (scanline processing) can change performance characteristics. If the debug output uses `println!` or file I/O inside a scanline loop, it changes the execution timing. Since the rendering is deterministic, this doesn't affect correctness -- but it affects development velocity. Each instrumented test run is 10-100x slower.

**Compounding mechanism:** Instrumentation code is write-once, maintain-forever. As the Rust code evolves (fixing bugs, refactoring), the instrumentation must be updated to match. If the instrumentation falls out of date, its output is silently wrong. Nobody notices because the instrumentation is only run when debugging, and when debugging, you assume the instrumentation is correct.

### 10. Rebuild Regression Risk

The 204 passing golden tests are the project's most valuable asset. Any approach that risks regressing them is playing with fire.

**The structural rewrite scenario.** Approach B may conclude that `paint_9slice_section` should be restructured to more closely match `ScanlineTool::Init`. This restructuring could change the computation order, variable naming, or intermediate rounding. Even if the new code is "more correct" (closer to C++), it may break currently-passing tests that depend on the existing Rust code's specific behavior.

**The "fix one, break three" pattern.** Border rendering involves 9 sections. Currently, some sections may be correct and others incorrect. If the restructuring changes the transform computation, it may fix the incorrect sections while changing the behavior of the correct ones. The net effect could be: 3 tests newly pass, 5 tests newly fail. The project is now at 202/241 instead of 204/241.

**The testing gap.** Golden tests are end-to-end. They don't tell you which of the 9 sections is correct and which is wrong. A restructuring that changes all 9 sections has its correctness verified only at the end-to-end level. There is no intermediate test that says "section 3 (bottom edge) is correct, section 7 (left edge) is wrong." Without this granularity, you cannot make targeted changes safely.

**Compounding mechanism:** Each regression creates pressure to revert. Each revert wastes the investigation time that led to the restructuring. After one revert, the team becomes risk-averse about structural changes. After two reverts, the team avoids structural changes entirely, which means Approach B's core mechanism (restructure to match C++) is foreclosed.

### 11. Scope Creep Through Infrastructure

Building intermediate comparison infrastructure is interesting engineering work. It involves designing data formats, writing serialization code, building diff visualization tools, and creating CI integration. Each component is individually justifiable.

**The expanding scope pattern.** Stage 1: "Compare section rectangles between C++ and Rust." This requires emitting section rectangles from both sides and comparing them. Stage 2: "Also compare transform parameters." This requires emitting more fields. Stage 3: "The transform parameters match but the scanline output differs -- compare individual scanlines." This requires emitting per-scanline interpolation buffers, which is 1024 bytes per scanline per section per frame. Stage 4: "The scanline at row 289 differs -- compare individual pixel sampling." This requires emitting per-pixel sampling coordinates and weights, which is thousands of values per frame.

At stage 4, the infrastructure is a differential rendering debugger. Building it took 3 investigation rounds. The root cause (say, an off-by-one in loop bounds) could have been found by staring at two 10-line code snippets for 30 minutes.

**Compounding mechanism:** Infrastructure investment creates commitment to using the infrastructure. If the infrastructure says "transforms match," the investigator trusts this and looks elsewhere. But if the infrastructure's comparison is at the wrong granularity or the wrong point in the pipeline, "transforms match" is a false negative. The infrastructure becomes a trusted oracle that may be systematically wrong, and its trustedness prevents questioning it.

### 12. The 12-Group Diversity Problem

The 37 failing tests span 12 groups. Approach B's pipeline decomposition targets image rendering (Groups A+B+D, 21 tests). The remaining groups involve:

- **G2** (2 tests): Polygon rasterization (different code path entirely -- `PaintPolygon`, not `PaintImage`).
- **G3** (2 tests): Composition/layering (compositor logic, not painter internals).
- **G5** (2 tests): Starfield rendering (custom panel, different code path).
- **G7** (2 tests): Gradient rendering (linear/radial gradient interpolation).
- **G8** (2 tests): Structural mismatches (widget layout, not rendering).
- **G9** (1 test): Polyline stroke (Bezier evaluation, not image rendering).

Building image-rendering pipeline comparison infrastructure helps with 21/37 tests. The remaining 16 tests require entirely different investigation approaches. The infrastructure investment yields zero benefit for these groups.

**The tail problem.** After Group A+B is fixed (whether by Approach A or B), there are still 16 tests in 8 groups. Each group may require its own investigation. The infrastructure built for Group A+B does not transfer. The project returns to Approach A's methodology for these remaining groups.

**Compounding mechanism:** The time invested in image-rendering infrastructure is time not invested in fixing the diverse tail groups. If the tail groups are individually easy (each a single investigation round), the infrastructure investment has high opportunity cost. If the tail groups are individually hard, the infrastructure investment is irrelevant to them. Either way, the infrastructure helps only the one group it was designed for.

## Part III: Cross-Cutting Failures (Both Approaches)

### 13. Agent Context Limits and C++ Macro Expansion

Both approaches rely on agents reading C++ code. The C++ rendering code uses `#include`-based template expansion. `emPainter_ScTlIntImg.cpp` (1612 lines) is `#include`d multiple times with different `#define` settings for `CHANNELS` (1,3,4), `EXTENSION` (edge, zero, clamp), producing 12 variants of each interpolation function. The actual function that runs for a specific test case is one of these 12 variants.

An agent reading the template sees the generic version with `#if CHANNELS>=4` guards. To understand the specific behavior for CHANNELS=4 (the common case for border images, which are RGBA), the agent must mentally expand the template. Agents do this unreliably. They may:
- Forget that a `#if CHANNELS==1` block is skipped.
- Misinterpret a `#if CHANNELS>=3` block as applying to CHANNELS=4 (correct) but forget the additional code in a `#if CHANNELS>=4` block.
- Confuse the extension mode variants (EDGE vs ZERO vs CLAMP have subtle differences in boundary handling).

Neither Approach A nor Approach B addresses this. It is an irreducible difficulty of the C++ source.

**Compounding mechanism:** Each agent session starts fresh and must re-learn the macro expansion. There is no way to "compile" the macro-expanded version and hand it to the agent (the expansion is context-dependent). Each session re-incurs the risk of macro misinterpretation.

### 14. Golden Test Oracle Reliability

The user has flagged: "Phase 5 golden tests may have wrong setup (fill, canvas, scale); verify against C++ generator before assuming rendering bugs." If some golden reference images are incorrect, achieving pixel parity with them is the wrong goal. But there is no mechanism to distinguish "reference image is correct, Rust diverges" from "reference image is wrong, Rust is actually correct."

**The scenario.** A test generates a scene with a border widget at a specific zoom level. The C++ generator creates the reference image. If the generator initializes the scene slightly differently from how the Rust test does (different canvas color, different clip rectangle, different painter state), the reference image records the C++ output for a different scene than the Rust test renders. Achieving pixel parity requires the Rust code to reproduce the *generator's* scene, not the intended scene. This is the wrong goal, and achieving it would actually introduce bugs.

**The "correct divergence" trap.** Some of the 37 failing tests may be failing because the Rust code is *more correct* than the C++ generator. If the Rust code correctly renders a widget that the C++ generator renders with a bug, fixing the "divergence" means introducing the C++ bug into the Rust code. This produces pixel parity at the cost of correctness.

**Compounding mechanism:** Neither approach has a way to detect this. Both use the golden reference as ground truth. If 5 of the 37 failures are actually Rust-correct/C++-wrong, both approaches will waste 5 investigation rounds trying to introduce bugs. The remaining 32 genuine divergences will be mixed with the 5 phantom divergences, and there is no way to tell them apart from the golden test output alone.

### 15. Diminishing Returns on Both Axes

The 204 passing tests represent the easy wins. The 37 remaining tests are the hard cases. As each successive fix closes more tests, the remaining tests become progressively harder.

**The easy/hard distribution is not uniform.** The first 20 of the 37 failures (HowTo text) were fixed by a single wiring change. The next 5 (blend_colored_scanline) were fixed by a single function port. The remaining 37 involve rendering pipeline internals. The next batch might be 7 tests fixed by a gradient coefficient. The batch after that might be 2 tests requiring deep area-sampling investigation. The final batch might be 1 test requiring a complete rewrite of polyline stroke.

**Compounding mechanism:** Both approaches estimate future effort based on past effort. But past effort is dominated by the easy cases (HowTo text, blend fix), which required 1 round each. Future effort will be dominated by the hard cases (area sampling, composition), which may require 3-5 rounds each. The past is not predictive of the future. Both approaches will underestimate remaining effort.

### 16. The "Correct but Different" Classification Problem

CLAUDE.md defines when literal C++ ports are required (pixel arithmetic, coverage, interpolation, sampling) and when idiomatic Rust is acceptable (state logic, data structures, ownership, API surface). But the boundary between these categories is not always clear.

**The gray zone.** `paint_9slice_section` computes sub-pixel edge positions. Is this "pixel arithmetic" (literal port required) or "geometry" (same algorithm and operation order, but idiomatic Rust OK)? If it's pixel arithmetic, every intermediate computation must match C++ exactly. If it's geometry, the algorithm and operation order must match but the expression-level implementation can use Rust idioms.

The C++ code computes: `ImgMap+=ImgDX*(t>>1)`. This is pointer arithmetic that adjusts the image map base for stride reduction. The Rust code doesn't use raw pointers. It computes an offset as an integer. Are these "the same algorithm"? They produce the same result if and only if the Rust integer computation produces the same offset as the C++ pointer arithmetic. But verifying this requires comparing the arithmetic, which requires understanding the C++ pointer arithmetic (which involves `sizeof` implications, signed/unsigned conversions, and platform-dependent pointer sizes).

**Compounding mechanism.** Every time an investigator encounters a gray-zone function, they must make a classification decision. These decisions are not recorded. The next investigator makes the same decision independently, possibly differently. Across multiple investigation rounds, inconsistent classifications accumulate. Some functions are treated as "literal port required" in one round and "idiomatic Rust OK" in another. This inconsistency means that some functions are investigated too deeply (wasting time on allowed differences) and others too shallowly (missing required exactness).

### 17. The Shared Failure Mode: No Behavioral Oracle Beyond Golden Tests

Both approaches ultimately rely on golden tests as the final arbiter. Intermediate harnesses (Approach B) add granularity but do not replace golden tests. Hypothesis elimination (Approach A) is guided by golden test failures.

**The problem.** Golden tests are black-box end-to-end tests. They say "the output image differs." They do not say "the area sampling loop computed a wrong weight at pixel (32, 289)." Both approaches must bridge the gap between the golden test's observation (pixel differs) and the root cause (wrong weight / wrong coordinate / wrong blend / wrong clip). This bridging is the hard part. Neither approach eliminates it.

**The "last mile" problem.** Even if Approach B's intermediate harnesses narrow the divergence to "the interpolation stage," the investigator must still read the C++ and Rust interpolation code side-by-side and find the difference. This is the same task that Approach A's investigators perform. The intermediate harness saved the backward-trace steps (items 4 above), but the final comparison step is identical. If the root cause is in a macro-expanded C++ template that the agent misreads, both approaches fail identically at this step.

**Compounding mechanism:** Both approaches spend most of their effort on the "not the last mile" part (tracing backward, building harnesses, eliminating hypotheses). The last mile -- the actual line-by-line comparison of C++ and Rust integer arithmetic -- is where both approaches converge and where both approaches are equally vulnerable to agent limitations. Optimizing everything except the last mile produces diminishing returns because the bottleneck is the last mile.

### 18. The Interaction Between Failing Groups Creates False Isolation

The 37 failures are grouped into 12 groups based on apparent similarity (same test file, same divergence pattern). But this grouping may be incorrect.

**The shared-root-cause scenario.** Group A+B (border image, 21 tests) and G7 (gradient, 2 tests) may share a root cause. If both are caused by a sub-pixel edge calculation error, fixing the error fixes both groups. But the groups are investigated separately because they look different (one involves images, the other involves gradients). The investigation of Group A+B never looks at gradient code, and the investigation of G7 never looks at image code. The shared sub-pixel edge code is in neither investigation's scope.

**The false-isolation mechanism.** Grouping tests by failure pattern creates the assumption that each group has its own root cause. This assumption is useful for triage but can be wrong. If two groups share a root cause, investigating them separately doubles the effort and halves the probability of finding the cause (because each investigation has a smaller search space that excludes the shared code).

**Compounding mechanism:** Both approaches use the group structure for prioritization. Approach A dispatches investigations per group. Approach B builds harnesses per pipeline stage. But the group structure is derived from symptoms (which tests fail, what the pixel diffs look like), not from causes (which code paths are exercised). If symptoms don't map cleanly to causes, the group structure misleads both approaches equally.

### 19. Temporal Instability of the Passing Set

The 204 passing tests pass at zero tolerance (exact pixel match). But this exactness is fragile.

**The "one commit away from regression" problem.** The Rust rendering code uses `f64` for geometry and `i32`/`i64` for pixel arithmetic. A refactoring that changes the order of floating-point operations (e.g., replacing `a * b + c` with `c + a * b`) can change the result by 1 ULP in the geometry, which changes the sub-pixel edge calculation by 1 sub-pixel unit, which changes one pixel's coverage by a small amount, which changes the pixel's color by 1 channel value. Zero-tolerance tests fail.

This can happen in either approach. Approach A's "fix the root cause" may involve a refactoring that changes float operation order. Approach B's "restructure to match C++" may involve reorganizing code in ways that change float operation order. Both approaches risk regressing currently-passing tests through mathematically equivalent transformations that are not bit-identical.

**Compounding mechanism:** Each fix or restructuring is tested against the golden suite. If it causes regressions, it must be reverted or adjusted. But the adjustment may introduce new regressions in a different subset of tests. This creates a whack-a-mole dynamic where fixing one test breaks another, not because the code is wrong, but because zero-tolerance golden tests are sensitive to non-semantic code changes.

### 20. The Unobserved Correctness Dimension: Sub-Pixel Positioning

Both approaches focus on pixel color values (the output of rendering). Neither approach systematically verifies sub-pixel positioning (the input to rendering). If a border image section's destination rectangle is off by 0.001 pixels, the section renders at a slightly different position. This shifts which pixels are edge pixels (partial coverage) and which are interior pixels (full coverage). The edge pixels change color by the coverage difference. The interior pixels are unaffected.

**The subtle signature.** The failing tests show divergences concentrated at specific pixel coordinates (e.g., x=31-40, y=288-289 in multiple widget tests). These are edge pixels. If the sub-pixel positioning differs between C++ and Rust, the edge pixels differ but the interior pixels match. This is consistent with all observations: small channel differences (max_diff=14), concentrated at specific coordinates, affecting a small percentage of pixels.

**Why neither approach checks this.** Approach A checks "do the transforms match" and concludes "yes" based on comparing transform parameters. But transform parameters determine how source pixels map to destination pixels, not where the destination rectangle starts. The destination rectangle's sub-pixel position is determined earlier, in the 9-slice subdivision logic. Approach B decomposes the pipeline into stages, but the stage boundaries might not include "sub-pixel edge calculation" as a distinct stage -- it's embedded in `paint_9slice_section`'s coordinate computation.

**Compounding mechanism:** Both approaches investigate downstream of the actual divergence. Each investigation concludes "this stage matches" because it does match -- the divergence is upstream. The upstream stage (sub-pixel positioning) is never investigated because it's embedded in "setup" code that both approaches treat as part of the calling context, not as a stage in its own right.
