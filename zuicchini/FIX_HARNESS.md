# Fix Harness

> Structured process for fixing verification findings.
> Consumes `_verify/` artifacts. Produces code changes and re-verified verdicts.

---

## Design Principles

The verification harness is skeptical by structure: it asks "does this
match?" and the answer is checkable. Fixing is the opposite — the agent
*wants* the code to work and will convince itself it does. This harness
counteracts that bias.

| Principle | Mechanism |
|---|---|
| **Context before action** | Phase 1 forces module-level understanding before any code is written |
| **Batch by type, not by finding** | All fixes run in one phase, all re-verification in another — not interleaved |
| **Module grouping** | Fixes within a module are applied together by one agent that holds full module context |
| **Separation of writer and judge** | The agent that writes a fix never judges whether it worked — a separate Phase 3 re-run does that |
| **Skip, don't guess** | Design decisions requiring human input are accumulated and presented as a batch — they never block the pipeline |
| **Regression breadth** | Re-verification covers the touched method, its neighbors, and previously-CORRECT methods in the same file |

## Paths

```
Inputs (read-only — the fix harness never modifies these):
  _verify/extracts/*.yaml       behavioral specs — what the code should do
  _verify/mappings/*.yaml       C++ to Rust mapping with status per method
  _verify/verdicts/*.yaml       branch-level findings with severity
  EMCORE_FEATURE_CONTRACT.md    intentional divergences

Outputs (generated by the fix harness):
  _fix/triage.yaml              classified, grouped findings
  _fix/design_decisions.yaml    items requiring human input
  _fix/context/<module>.yaml    per-module understanding docs
  _fix/changes/<module>.yaml    per-module change logs
  _fix/baselines/               pre-fix verdict snapshots

Source:
  zuicchini/src/                Rust source (modified by Phase 2)
```

## Severity Definitions

These severities appear in verdict files and are used throughout the
fix harness. Defined here so all agents share the same vocabulary.

| Severity | Meaning |
|---|---|
| **CORRECT** | Rust matches the behavioral spec |
| **COSMETIC** | Differs in naming or style but behavior is equivalent |
| **BEHAVIORAL** | Logic diverges in a way that changes output for some inputs |
| **CRITICAL** | Branch is entirely missing or inverted |

---

## Fix Categories

| Category | Description | Autonomy | Example |
|---|---|---|---|
| **BUG** | Wrong behavior in existing Rust code | Fully autonomous | Double undo on type-over |
| **MISSING_BRANCH** | Method exists but a conditional path is absent | Fully autonomous | No min-dimension clamp |
| **MISSING_METHOD** | C++ method with no Rust counterpart | Autonomous (port from extract) | PaintBezier |
| **DESIGN** | Behavioral divergence that may be intentional | Human-in-the-loop | Layout default prefCT 1.0 vs 0.2 |

## Phases

### Phase 0: TRIAGE — Classify and Group Findings

**Input:** All verdict files, all mapping files.
**Output:** `_fix/triage.yaml`, `_fix/design_decisions.yaml`
**Autonomy:** Fully autonomous.

One agent reads all verdict and mapping files. For each CRITICAL or
BEHAVIORAL finding and each MISSING mapping entry, it produces a
classified, grouped list.

**Grouping rules:**
- Group by Rust module (use the `rust_file` field from the mapping)
- Within a module, order by dependency (if method A calls method B,
  fix B first)
- Tag each finding with its category (BUG / MISSING_BRANCH /
  MISSING_METHOD / DESIGN)
- DESIGN items go into `_fix/design_decisions.yaml` — they do not
  enter Phase 1 or Phase 2

**Output schema:**

```yaml
modules:
  - rust_file: <path>
    fixes:
      - id: <module>-<n>
        category: BUG | MISSING_BRANCH | MISSING_METHOD
        source_finding: <verdict file>:<method>:<condition>
        extract_ref: <extract file>:<method>
        cpp_lines: <start>-<end>     # for MISSING_METHOD only
        dependency: [<id>, ...]      # fixes in this module that must come first
        summary: <one line>

design_decisions:
  - id: <module>-D<n>
    source_finding: <verdict or mapping file>:<method>
    skip_reason: <why this cannot be fixed autonomously — must be specific>
    question: <what needs human input>
    options:
      - <option A and its consequence>
      - <option B and its consequence>
```

**Ambiguity safeguard:** Err toward DESIGN. A DESIGN item that turns
out to be a straightforward fix wastes one human decision; a BUG item
that turns out to be a design choice silently embeds the wrong answer.

**Cross-module dependencies:** After grouping, scan for fixes that
change a public API. For each, check whether other modules call that
API. Record dependencies:

```yaml
cross_module_dependencies:
  - upstream_fix: <fix id in module A>
    downstream_modules: [<module B>, <module C>]
    reason: <what changes and who is affected>
```

Modules with upstream dependencies run after their upstream module
in Phase 2 (not in parallel).

**Agent prompt skeleton:**

```
You are triaging verification findings for a Rust reimplementation of
C++ emCore. Your job is to classify every finding and group them by
Rust module for fixing.

Read ALL verdict files in: zuicchini/_verify/verdicts/*.yaml
Read ALL mapping files in: zuicchini/_verify/mappings/*.yaml

The mapping files tell you which Rust file and function corresponds
to each C++ method. Use the rust_file field to group findings by
Rust module. Use the rust_fn and rust_lines fields to identify the
specific Rust function for each finding.

For every entry with severity CRITICAL or BEHAVIORAL in the verdicts,
and every entry with status MISSING in the mappings, create a triage
entry.

CLASSIFICATION DECISION TREE — apply in order:

1. Does the Rust method exist?
   NO  → Is this method referenced or needed by existing Rust code?
         YES → MISSING_METHOD
         NO  → DESIGN (skip_reason: "method has no callers in Rust;
                need human decision on whether to port it")
   YES → continue to question 2

2. Is the correct behavior unambiguous from the extract alone?
   Could a developer read the extract and produce exactly one correct
   implementation, with no judgment calls?
   NO  → DESIGN (skip_reason: explain what is ambiguous)
   YES → continue to question 3

3. Does the Rust code have wrong behavior, or is it missing a branch?
   Wrong behavior (inverted condition, wrong formula, wrong constant
   where the correct value is stated in the extract) → BUG
   Missing branch (a conditional path from the extract is absent)
   → MISSING_BRANCH

EXAMPLES — classify as DESIGN:

- A constant differs (C++ uses 0.2, Rust uses 1.0) and the extract
  just states the C++ value. → DESIGN
  skip_reason: "Rust uses 1.0 vs C++ 0.2; may be intentional"

- C++ uses algorithm A (log-error scoring), Rust uses algorithm B
  (linear scoring). Both produce valid results. → DESIGN
  skip_reason: "different algorithm, both plausible"

- A MISSING method that nothing in the Rust codebase calls. → DESIGN
  skip_reason: "no callers; may not be needed for egopol"

EXAMPLES — classify as BUG or MISSING_BRANCH:

- Extract says "save_undo once" but Rust calls save_undo twice.
  Only one correct behavior. → BUG

- Extract says "clamp min dimension to prevent div-by-zero" and
  Rust has no clamp. A guard, not a design choice. → MISSING_BRANCH

- Extract says "if alpha == 0, skip blending" and Rust blends
  anyway (causes incorrect darkening with canvas_blend). → BUG

Group by Rust module (use rust_file from the mapping). Within each
module, determine dependency order by reading the Rust source and
checking which methods call which. For MISSING_METHOD fixes, check
the extract's calls field to see if the missing method would call
another method that also has a fix.

After grouping, scan for cross-module dependencies: any fix that
adds, removes, or changes a public method signature. For
MISSING_METHOD, every new public method is a potential cross-module
dependency — check if any other module references the name. Record
in cross_module_dependencies.

Write triage to: zuicchini/_fix/triage.yaml
Write design decisions to: zuicchini/_fix/design_decisions.yaml

Quality gate — run and report the result:
  Count CRITICAL + BEHAVIORAL in verdicts (grep -c)
  Count entries in triage.yaml + design_decisions.yaml
  The counts must match.
```

**Quality gate:** Every CRITICAL and BEHAVIORAL finding appears in
either `triage.yaml` or `design_decisions.yaml`. Enforce mechanically:
`grep -c 'severity: CRITICAL\|severity: BEHAVIORAL'` across all
verdict files, compare against entry count in triage + design_decisions.

### Phase 1: CONTEXT — Understand Before Changing

**Input:** `_fix/triage.yaml`, Rust source, extracts, mappings.
**Output:** `_fix/context/<module>.yaml`
**Autonomy:** Fully autonomous (read-only — no code changes).

One agent per Rust module. The agent reads the Rust source file, the
behavioral extract, the mapping file, and the triage entries for its
module. It produces a context document:

```yaml
module: <rust file>
patterns:
  - <conventions: naming, error handling, coordinate system,
    ownership patterns, state management>
file_structure:
  - <how the file is organized: method grouping, ordering>
callers:
  - method: <method being fixed>
    called_by: [<call sites found via grep>]
    calls: [<methods this function calls>]
integration_notes:
  - <what the fixing agent needs to know to produce code that fits>
```

**Agent prompt skeleton:**

```
You are analyzing a Rust module to understand its patterns before
any code changes are made. You will NOT modify any files.

Read the following:
1. Rust source: {rust_file_path}
2. Behavioral extract: {extract_file_path}
3. Mapping file: {mapping_file_path}
   (Use rust_fn and rust_lines to find the Rust function for each
   C++ method. Use RESTRUCTURED notes to understand how C++ concepts
   were translated to Rust.)
4. Triage entries for this module: {triage_entries}

Produce a context document that answers:
- What naming conventions does this module use?
- What coordinate system / units? (pixels, floats, user-space, etc.)
- What ownership pattern? (&self, &mut self, takes owned values?)
- How is state managed? (struct fields, push/pop stack, etc.)
- How does error handling work? (Result, Option, panic, silent skip?)
- How is the file organized? (method grouping — e.g., "constructors
  at top, paint methods lines 120-400, helpers at bottom")

For each fix ID in the triage:
- Use the mapping file to find the Rust function name
- Find all call sites (grep zuicchini/src/ for the function name)
- Find all methods that function calls
- Note anything the fixing agent needs to produce code that fits

For MISSING_METHOD fixes:
- The method does not exist yet — there are no callers to find.
  Instead note: where in the file should it be added? What existing
  methods are most similar (for style reference)? What do the
  mapping's RESTRUCTURED notes suggest about Rust-specific adaptation?

Write output to: {context_output_path}
Do NOT modify any source files.
```

**Quality gate:** Every fix ID appears in the `callers` section with
at least one caller or an explicit note that it is a new entry point.

### Phase 2: FIX — Apply Changes

**Input:** Triage, context doc, extract, mapping, Rust source.
**Output:** Modified Rust source, `_fix/changes/<module>.yaml`
**Autonomy:** Fully autonomous for BUG, MISSING_BRANCH, MISSING_METHOD.

One agent per module. The agent receives:
1. The triage entries for this module (fixes to apply)
2. The context document (patterns to follow)
3. The behavioral extract (spec for correct behavior)
4. The mapping file (C++ to Rust name translation)
5. The current Rust source (code to modify)

**Rules:**

- **Follow the context document.** Match existing naming, coordinate
  conventions, ownership patterns, and abstraction level.

- **Use the extract as the spec.** Implement what it says, adapted
  to the Rust module's patterns as described in the context document.

- **One module at a time.** Apply all fixes in dependency order from
  the triage. If the module has more than 20 fixes, split into chunks
  of 10-15. Each chunk is a separate agent pass. Later chunks read
  the modified source from earlier chunks. The change log accumulates.

- **No scope creep.** Do not refactor, rename, reformat, add comments,
  or "improve" code outside the fix.

- **Record what changed:**

```yaml
module: <rust file>
changes:
  - fix_id: <from triage>
    action: MODIFIED | ADDED | REPLACED | SKIPPED_DESIGN | BLOCKED | RECLASSIFIED
    lines_before: <start>-<end>  # for MODIFIED/REPLACED
    lines_after: <start>-<end>   # omit for SKIPPED/BLOCKED/RECLASSIFIED
    description: <what was changed and why>
    reason: <for SKIPPED/BLOCKED/RECLASSIFIED: why it was skipped>
    extract_branches_addressed:
      - <list of extract branch conditions this fix satisfies>
```

**Agent prompt skeleton:**

```
You are fixing verified gaps in a Rust reimplementation of C++ emCore.

Read the following:
1. Triage entries for this module: {triage_entries}
2. Context document: {context_file_path}
3. Behavioral extract (your spec): {extract_file_path}
4. Mapping file: {mapping_file_path}
   (The extract uses C++ names like emColor, PaintPolygon. The mapping
   tells you the Rust equivalents: Color, paint_polygon. RESTRUCTURED
   notes explain how C++ patterns were adapted.)
5. Current Rust source: {rust_file_path}

Rules:
- Apply fixes in the dependency order listed in the triage.
- The extract defines correct behavior. Implement what it says,
  adapted to the patterns described in the context document.
- Do NOT refactor, rename, reformat, add comments, or change code
  outside the scope of the listed fixes.
- If a fix depends on an unresolved DESIGN decision, skip it and
  record it as SKIPPED_DESIGN in the change log.
- If a fix won't compile after 3 attempts, revert it and record
  it as BLOCKED with the compilation error.
- RECLASSIFICATION: If while implementing a fix you discover it
  involves a judgment call — the "correct" value could go either
  way, or the fix would change behavior that might be intentionally
  different — do NOT guess. Record it as RECLASSIFIED in the change
  log with a reason, skip it, and move on.

After all fixes, run: cargo check --workspace
If it fails, fix compilation errors in the files you modified only.

Write change log to: {changes_output_path}
```

**Quality gate:** `cargo check --workspace` passes. If compilation
cannot be resolved within 3 attempts, the agent reverts the failing
fix, marks it BLOCKED, and continues. BLOCKED and RECLASSIFIED fixes
surface in Phase 4.

### Phase 3: RE-VERIFY — Separate Agent Judges the Fixes

**Input:** Extracts, current Rust source.
**Output:** Updated verdict files in `_verify/verdicts/`.
**Autonomy:** Fully autonomous.

The re-verifying agent compares the behavioral extract against the
current Rust source. It does NOT see the change log, fix descriptions,
or any Phase 2 output. It judges the code as-is.

**Scope:** The orchestrator determines which methods to re-verify by
reading the change log (the agent never sees this). The scope is:
1. Every method modified or added in Phase 2
2. Every method in the same file that was previously CORRECT
   (regression check)
3. Every method that calls or is called by a changed method
   (integration check)

**Agent prompt skeleton:**

```
You are verifying a Rust reimplementation against a behavioral
specification. You will NOT see any change logs or fix descriptions.
You work only from the extract and the current Rust source.

Your default stance is skeptical. If you cannot find clear evidence
that a branch is handled, mark it NO. When uncertain between CORRECT
and BEHAVIORAL, choose BEHAVIORAL.

Read the following:
1. Behavioral extract (specification): {extract_file_path}
2. Current Rust source: {rust_file_path}

Verify ONLY these methods: {method_list}

For each method, check every branch, constant, and edge case from
the extract against the Rust code. Produce a fresh verdict — do not
assume any previous state.

Output format for each branch:

  - condition: <the condition from the extract>
    rust_handles: YES | NO | PARTIAL
    evidence: <quote specific Rust line(s), including file:line_number>
    severity: CORRECT | COSMETIC | BEHAVIORAL | CRITICAL
      CORRECT:    Rust matches the spec
      COSMETIC:   Differs in naming/style but behavior is equivalent
      BEHAVIORAL: Logic diverges for some inputs
      CRITICAL:   Branch is entirely missing or inverted

For each constant:
  - constant: <name>
    spec_value: <from extract>
    rust_value: <from Rust code, or MISSING>
    match: YES | NO | APPROXIMATE

For each edge case:
  - edge_case: <description>
    rust_handles: YES | NO
    evidence: <how, or what is missing>

If ALL branches, constants, and edge cases for a method are CORRECT:
  - method: <name>
    all_correct: true

Write output to: {verdict_file_path}
```

**Verdict snapshots:** Before updating verdict files, the orchestrator
copies the previous version to
`_fix/baselines/<verdict_file>.<round_number>.yaml`.

**Post-processing (orchestrator, not agent):**
After the agent produces verdicts, the orchestrator:
- Diffs against the baseline to tag new findings as REGRESSION
- Adds fix ID notes to findings that changed to CORRECT
- Flags findings that persist despite a fix being attempted

**Quality gate:** Checked per-severity against the baseline snapshot.
CRITICAL count must not increase. BEHAVIORAL count must not increase.
If either increases, the offending fix must be reverted or corrected.

---

### Phase 4: HUMAN REVIEW — Design Decisions

**Input:** `_fix/design_decisions.yaml`, BLOCKED and RECLASSIFIED
entries from change logs.
**Output:** Human-annotated decisions.
**Autonomy:** Human-in-the-loop.

Present all items requiring human judgment as a single batch:
- DESIGN items from triage (with `skip_reason`)
- BLOCKED fixes from Phase 2 (with compilation error)
- RECLASSIFIED fixes from Phase 2 (with reason)

The human annotates each with a decision. Decisions that result in
code changes re-enter Phase 2 as a new batch.

## Execution Model

The fix harness runs as a series of Claude Code agent delegations,
orchestrated by the main context.

```
Phase 0: Triage (one agent, all findings)
    |
Phase 1: Context (parallel, one agent per module, max 3)
    |
Phase 2: Fix (parallel for independent modules, sequential for
    |        cross-module deps, max 3)
    +-- cargo check --workspace
    |
Phase 3: Re-verify (parallel, one agent per module, max 3)
    |
    +-- regression gate
    |
Phase 4: Human review (batch of design decisions)
    |
    +-- if decisions produce fixes, loop to Phase 2
```

Multiple rounds are expected. Each round fixes autonomous items,
re-verifies, then presents design decisions. Loop until no new
fixes result from decisions.

**Agent naming convention:**
```
Triage:     "Triage all verification findings"
Context:    "Context for render/painter.rs"
Fix:        "Fix render/painter.rs (12 findings)"
Re-verify:  "Re-verify emPainter.cpp verdicts"
```

**Orchestrator responsibilities between phases:**
- After Phase 0: Verify the quality gate count match
- After Phase 1: Spot-check one context document for accuracy
- Before Phase 2: Resolve template variables in the fix agent prompt
  (fill in file paths from triage and mapping files)
- After Phase 2: Run `cargo check --workspace`
- Before Phase 3: Snapshot baseline verdicts. Extract the method list
  from change logs (the re-verify agent must NOT see the change logs
  — only the method list)
- After Phase 3: Post-process verdicts (diff baselines, tag
  REGRESSIONs, add fix ID notes). Run regression gate.

## Example

A finding flowing through all phases.

**Source finding** (from `_verify/verdicts/emTextField.yaml`):
```yaml
- method: emTextField::TextInput
  branches:
    - condition: "if selection exists, save_undo then delete_selection before inserting"
      rust_handles: PARTIAL
      evidence: |
        text_field.rs:231 calls save_undo(), then line 232 calls
        delete_selection() which also calls save_undo(). Double undo.
      severity: CRITICAL
```

**Phase 0 — triage entry:**
```yaml
- id: text_field-1
  category: BUG
  source_finding: emTextField.yaml:emTextField::TextInput:"if selection exists..."
  extract_ref: emTextField.yaml:emTextField::TextInput
  dependency: []
  summary: Double save_undo on type-over selection
```

**Phase 1 — context entry (excerpt):**
```yaml
callers:
  - method: text_input
    called_by: [widget/text_field.rs:handle_key_event (line 180)]
    calls: [save_undo, delete_selection, insert_text]
integration_notes:
  - "delete_selection() calls save_undo() internally (line 285).
     Fix must remove the outer save_undo(), not the inner one."
```

**Phase 2 — change log entry:**
```yaml
- fix_id: text_field-1
  action: MODIFIED
  lines_before: 229-233
  lines_after: 229-232
  description: >
    Removed save_undo() at line 231. delete_selection() already
    saves undo internally. Type-over now creates one undo entry.
  extract_branches_addressed:
    - "if selection exists, save_undo then delete_selection before inserting"
```

**Phase 3 — agent produces fresh verdict:**
```yaml
- method: emTextField::TextInput
  branches:
    - condition: "if selection exists, save_undo then delete_selection before inserting"
      rust_handles: YES
      evidence: "text_field.rs:229-232 calls delete_selection() (saves undo internally) then inserts"
      severity: CORRECT
```

**Orchestrator post-processing** adds `note: "Fixed by text_field-1"`
by diffing against the baseline.

## Running a Fix Pass

To run the full fix harness from a fresh context:

```
0. Verify prerequisites:
   ls zuicchini/_verify/verdicts/*.yaml    # must exist
   ls zuicchini/_verify/mappings/*.yaml    # must exist
   ls zuicchini/_verify/extracts/*.yaml    # must exist
   mkdir -p zuicchini/_fix/context zuicchini/_fix/changes zuicchini/_fix/baselines

1. Triage (Phase 0):
   Spawn one agent with the Phase 0 prompt.
   Verify: grep -c count match between verdicts and triage + design_decisions.

2. Context (Phase 1):
   For each module in triage.yaml, spawn a context agent (max 3 parallel).
   Verify: every fix ID appears in a context document.

3. Fix (Phase 2):
   For each module, spawn a fix agent (parallel for independent, sequential
   for cross-deps). Each reads triage + context + extract + mapping + source.
   Run: cargo check --workspace
   Note any BLOCKED or RECLASSIFIED fixes for Phase 4.

4. Re-verify (Phase 3):
   Snapshot verdicts to _fix/baselines/.
   For each changed module, spawn a re-verify agent (max 3 parallel).
   Each reads extract + current Rust source only (NOT the change log).
   Post-process: diff baselines, tag REGRESSIONs, add fix ID notes.
   Run regression gate:
     grep -c 'severity: CRITICAL' before vs after (must not increase)
     grep -c 'severity: BEHAVIORAL' before vs after (must not increase)

5. Human review (Phase 4):
   Present design_decisions.yaml + BLOCKED + RECLASSIFIED entries.
   Human annotates decisions.
   If decisions produce new fixes, loop to step 3 with a new triage.
```

## Quality Gates

| Phase | Gate |
|---|---|
| 0 (Triage) | Every CRITICAL/BEHAVIORAL finding classified. None dropped. |
| 1 (Context) | Every fix ID has caller/callee analysis. |
| 2 (Fix) | `cargo check --workspace` passes. No DESIGN items attempted. BLOCKED fixes escalated. |
| 3 (Re-verify) | CRITICAL and BEHAVIORAL counts each non-increasing vs baseline. |
| 4 (Human) | All DESIGN, BLOCKED, and RECLASSIFIED items presented. None auto-decided. |
